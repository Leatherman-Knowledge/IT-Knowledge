## Overview

Phase 12.1.1 covers all work completed after the initial Phase 12.1 implementation. It focused on fixing critical bugs discovered during testing, building a value resolution trace system for debugging inheritance, improving data freshness across scope changes, and adding URL-persisted channel/locale scope for deep linking.

---

## 1. Locale Validation — Language-Only Codes

### Problem
The locale code validation regex on both frontend and backend rejected two-character language-only codes like `en`, `de`, `fr`. Only BCP 47 codes with region subtags (e.g., `en-US`, `de-DE`) were accepted. This made it impossible to create base locales that serve as fallback roots.

### Changes
- **Frontend** (`client/src/pages/locales.tsx`): Updated the Zod form schema regex to optionally accept `^[a-z]{2}$` in addition to `^[a-z]{2}-[A-Z]{2}$`.
- **Backend** (`shared/validation.ts`): Updated the locale code validation regex to match the same relaxed pattern.
- **Error messages**: Updated to indicate that both formats are valid (e.g., "en" or "en-US").

### Alignment
- **Phase 03 (Locales):** Locale codes may be language-only (`en`, `de`) or BCP 47 with region (`en-US`, `de-DE`). Documentation and validation must accept both.

## 2. Product Editing State Reset on Navigation

### Problem
When navigating between products (e.g., clicking from a variant to its parent), stale editing state from the previous product persisted — edited values, field errors, override tracking, and channel/locale selections leaked across products.

### Changes
- Added a `useEffect` in `product-edit.tsx` that watches the `identifier` parameter and resets all editing state when it changes:
  - `editedValues` → empty Map
  - `overriddenAttrs` → empty Set
  - `fieldErrors` → empty Map
  - `selectedChannel` / `selectedLocale` → null (later preserved from URL when URL scope was added)
  - `activeTab` → "attributes"
- Uses `prevIdentifierRef` to detect when the identifier actually changes vs. initial mount.

---

## 3. Attribute Override Reset

### Problem
Once a user overrode an inherited attribute value on a child product, there was no way to remove that override and revert to the inherited value.

### Changes
- **Backend**: Added `DELETE /api/v1/products/:identifier/values/:attributeCode` endpoint that removes the stored value for a specific attribute+channel+locale scope, effectively restoring inheritance.
- **Frontend**: Added a "Reset" button (with `RotateCcw` icon) on overridden attributes that calls the delete endpoint and invalidates the values cache. After reset, the attribute reverts to showing the inherited value with its source indicator.

### Alignment
- **Phase 09 (Product Attribute Values):** Single-value delete for override reset is part of the values API surface. Document in 09 as an alternative to bulk delete.

---

## 4. Override Statistics Badge

### Problem
When viewing a parent product (model or sub-model), there was no indication of which attributes had been overridden by child variants. Users had no way to know at a glance whether children had diverged from inherited values.

### Changes
- **Backend**: Added `GET /api/v1/products/:identifier/override-stats` endpoint that returns a count of how many descendants have overridden each attribute.
- **Frontend**: Added a `GitBranch` badge next to attribute labels showing "N overrides" when descendants have stored their own values for that attribute. Badge only appears on model/sub-model products (not on leaf variants).

## 5. Variant Definition Axis Display Fix

### Problem
The variant definition axes were not displaying correctly in the product edit UI. The axis attribute codes were not being extracted properly from the variant definition structure.

### Changes
- Fixed parsing of the `axes` array structure from the variant definition API response: `axes: [{ level, attributes: [{code, type}] }]`.
- Corrected the extraction of axis codes at each level for proper display and filtering.

## 6. Recursive Descendant Completeness

### Problem
Completeness calculations only considered direct children, not the full descendant tree. A model with sub-models containing variants would not include variant-level completeness in the rollup.

### Changes
- Updated `calculateCompletenessOverview` in `server/lib/completeness.ts` to recursively traverse all descendants (sub-models and their variants) when building the per-variant breakdown.
- The completeness API response now includes completeness data for every product in the hierarchy, not just direct children.

### Alignment
- **Phase 11 (Completeness):** Completeness overview must recursively include all descendants, not only direct children.

---

## 7. Category Assignment Independence

### Problem
Products without any category assignments would throw errors or show broken UI when trying to calculate completeness or display the edit page.

### Changes
- Made category assignments fully optional — products can exist and be edited without belonging to any category.
- Completeness calculation no longer fails when a product has no categories.
- The edit page gracefully shows an empty categories tab.

### Alignment
- **Phase 10 / 12:** Categories are optional. Product edit and completeness must handle zero categories.

## 8. Variant Definition Edit Protection

### Problem
Users could edit a variant definition (changing axes, levels) even when products already existed using that definition, which would corrupt the hierarchy structure.

### Changes
- Added a check that prevents editing variant definitions that are referenced by existing products.
- The UI disables the edit controls and shows a message explaining why editing is blocked.

### Alignment
- **Phase 4.5 (Variant Definitions):** Edit is blocked when the definition is in use by products.

## 9. Attribute Level Migration

### Problem
When moving an attribute to a different level in a variant definition (e.g., from level 0 to level 1), existing values stored at the old level became orphaned and inaccessible.

### Changes
- Added logic to migrate existing attribute values when an attribute's level assignment changes.
- Values are moved to the appropriate products in the hierarchy based on the new level.

### Alignment
- **Phase 4.5 / 09:** Variant definition updates that change attribute level must migrate existing values to the correct level.

## 10. Axis Attribute Display Filtering

### Problem
Axis attributes (the attributes that define variant identity, e.g. "color", "size") were appearing in the main attribute editing area even though they are immutable on variants. They showed as editable but saves would fail.

### Changes
- Filtered axis attributes out of the editable attribute groups in the product edit UI.
- Axis values are still displayed as read-only badges in the product header area.

### Alignment
- **Phase 12 (Product UI):** Editable attributes list must exclude axis attributes; axis values shown only as read-only in header.

## 11. Locale Fallback Inheritance Indicators

### Problem
When a value was resolved via locale fallback (e.g., viewing `de-DE` but the value came from `de` or `en`), the UI showed it the same as a direct value with no indication of its source.

### Changes
- Extended the `isAttrInherited()` helper to detect locale fallback values (where `source.fromLocale` is set and differs from the requested locale).
- Both hierarchy inheritance (`source.type === "inherited"`) and locale fallback (`source.fromLocale` set) are now treated as "inherited" in the UI.
- Source badges show the origin: "Inherited from PRD-00003" for hierarchy, "From locale en" for locale fallback, or both combined.

### Alignment
- **Phase 12.1:** UI must show locale fallback source (fromLocale) in addition to hierarchy inheritance (fromIdentifier).

## 12. Value Resolution Trace Modal

### Problem
With multi-dimensional inheritance (hierarchy × locale fallback), it was difficult for users to understand exactly where a value came from and why. The source badges gave a summary but not the full resolution path.

### Changes

**Backend** — `GET /api/v1/products/:identifier/values/trace/:attributeCode`
- Query params: `channelCode`, `localeCode`
- Traces the full resolution path step by step: for each locale in the fallback chain, for each product in the hierarchy (self → parent → … → root).
- Records whether a value was found, what it was, and why.
- Returns an ordered array of resolution steps with a `winner` flag on the step that provided the effective value.
- Each step includes: `productIdentifier`, `productLevel`, `localeCode`, `channelCode`, `value`, `reason` (e.g. "Direct value", "Locale fallback", "Inherited from parent").

**Frontend** — `ValueResolutionModal` component
- Triggered by clicking inherited-source badges or "Trace" badges on attributes.
- Displays the resolution steps in a visual timeline; highlights the winning step.
- Shows the full value at each step (or "No value" placeholders).
- Links to ancestor products are clickable for navigation.

### Alignment
- **Phase 12 (Product UI):** Value resolution trace is part of the product edit UX for inherited/fallback values. Phase 09 may reference the trace endpoint in API docs.

## 13. Forced Data Freshness on Scope Change

### Problem
When switching channel or locale in the product edit page, TanStack Query would serve cached data from the previous scope, causing the UI to show stale values until a manual refresh. The `editedValues` map was only synced once on initial load and didn't update when the scope changed.

### Changes
- **Query staleness:** Set `staleTime: 0` on all product-specific queries in `product-edit.tsx` (product details, values, completeness, categories, channels, override-stats). ~~Global reference data (attributes, families, locales, etc.) originally retained `staleTime: Infinity`, but this was later changed~~ — the global TanStack Query default `staleTime` was changed from `Infinity` to `0` so that all queries refetch fresh data on page navigation (fixing stale data after edits on other pages). Individual queries may still set explicit overrides where appropriate (e.g. `staleTime: 60000` for inline asset thumbnails).
- **Edited values sync:** Introduced `lastSyncedScopeRef` tracking the `valuesQueryString` (e.g. `includeInherited=true&channelCode=ecommerce&localeCode=en-US`) that was last synced to `editedValues`. The sync effect re-runs whenever `valuesQueryString` changes AND data is available; skips when `productValues.length === 0` to avoid premature sync during loading. Resets to `""` when identifier changes.

### Alignment
- **Phase 12 (Product UI):** Product edit must use fresh data on channel/locale change; sync edited state only when product values for the current scope have loaded (length > 0).

## 14. Critical Locale Display Bug Fix

### Problem
After implementing scope change data freshness, a race condition caused the wrong locale's data to display. The sync effect would fire during the loading state (when `productValues` was empty from the previous query being invalidated), marking the new scope as "synced" before actual data arrived. When data finally loaded, the effect wouldn't re-run because the scope ref already matched.

### Changes
- Guard: the sync effect only runs when `productValues.length > 0`.
- The `lastSyncedScopeRef` comparison ensures the effect fires again when real data arrives for the new scope.
- This fixed the issue where switching from `en-US` to `de-DE` would show `en-US` values under the `de-DE` label.

## 15. URL-Persisted Channel/Locale Scope

### Problem
When navigating between products (clicking a variant, a breadcrumb ancestor, or a completeness breakdown link), the selected channel and locale would reset to defaults. Users had to re-select their scope every time. There was no way to share a direct link to a specific product in a specific scope.

### Changes
- **URL query parameters:** Channel and locale selection are stored as URL query parameters: `/products/:identifier?channel=ecommerce&locale=en-DE`. Uses `wouter`'s `useSearch` to read query params; uses `history.replaceState` to update the URL silently when the user changes scope via the picker.
- **Scope preservation across navigation:** All navigation links include the current scope: breadcrumb ancestor links, variant row links, sub-model accordion links, categories tab "managed on root product" link, completeness breakdown variant click handler. `scopeQuery` prop is threaded through `VariantsTab` → `OneAxisView` / `TwoAxisView` → `VariantRow` / `SubModelAccordion`, and added to `CategoriesTab`.
- **Deep linking:** Visiting a URL with `?channel=X&locale=Y` initializes state from URL params; navigating to a different product with scope params already in the URL preserves channel/locale.

### Alignment
- **Phase 12 (Product UI):** Product edit route should support `?channel=&locale=` for deep linking and scope preservation. All in-page links to other products must append current scope query.

---

## 16. Channel Fallback (Deferred)

### Decision
Evaluated adding `fallbackChannelCode` to the channels table to allow one channel to inherit attribute values from another (mirroring locale fallback). **Deferred** because:
- Resolution already has two fallback dimensions (hierarchy × locale); a third adds significant complexity.
- Architecture supports it (e.g. `source.fromChannel` in types); adding later is straightforward (estimated one day) and requires no refactor.
- Waiting allows real-world usage to inform the design.

## Current System State (Reference)

### Value Resolution Order
1. For each locale in the fallback chain (requested locale → fallback → …)
2. For each product in the hierarchy (self → parent → … → root)
3. Check for a value matching the locale and channel
4. First non-empty value wins

### Source Annotation Structure
```typescript
source: {
  type: "own" | "inherited";
  fromIdentifier: string;
  fromLevel: number;
  fromLocale?: string;    // present when value came via locale fallback
  fromChannel?: string;   // reserved for future channel fallback
}
```

### Key Architecture Patterns
- `lastSyncedScopeRef` tracks the last synced `valuesQueryString` to prevent race conditions during scope changes.
- Global `staleTime: 0` default on all queries (changed from `Infinity`); individual queries may override (e.g. asset thumbnails at 60s).
- URL query params (`channel`, `locale`) drive scope selection; updated via `history.replaceState`.
- `scopeQuery` string threaded through component tree for link generation.
