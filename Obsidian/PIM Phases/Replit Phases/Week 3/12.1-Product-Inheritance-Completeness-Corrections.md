## Objective

Three correction areas that Replit’s current implementation gets wrong or omits. These become the canonical behavior for inheritance, UI display of fallback values, and completeness (including locale “completeness reset” and interactive scoring).

## 1. Override at Any Level & First-Available Value

### Requirement

- **Override:** Any product at any level below the attribute’s “owner” level may **override** a parent’s attribute value (except variant **axis** attributes, which are immutable).
- **Inheritance (read):** Resolve the **first available value** by walking (1) **hierarchy** (self → parent → grandparent → … root), and (2) **locale fallback** and **channel fallback** (where implemented). Combine these so that the effective value is the first non-empty result from that combined resolution.
- This must work **in unison** with existing channel and locale fallback logic (Phase 03).

### Current Wrong Behavior

- Phase 09 currently **rejects** writing a level-0 attribute on a level-2 variant (“that value must be written on the root model”). So overrides at a lower level are not allowed.
- Read path may not consistently apply “first available” across hierarchy + locale + channel in one unified pass.

### Correct Behavior

**Write path (API & services):**

- Allow writing an attribute on a product if:
  - **(a)** The product’s level is the attribute’s assigned level in the variant definition (normal case), **or**
  - **(b)** The product is a **descendant** of the level that owns the attribute (override case). Any level below the owner may store an override.
- **Axis attributes** remain read-only after creation; no override.
- Validation (family, scoping, type, uniqueness) unchanged; only the “level check” is relaxed as above.

**Read path (resolution order):**

1. For the requested (channel, locale) scope, resolve with **locale fallback** and **channel fallback** (if any) at each level.
2. Walk **hierarchy** from current product up to root. At each level, for the attribute, try (channel, locale) then fallbacks. The **first** non-empty value in that walk is the resolved value.
3. Annotate each value with **source**: own / inherited (and from which product), and if from fallback, which locale/channel it came from.

So: “First available value” = first found when combining hierarchy walk with locale (and channel) fallback at each step. No “level ownership” restriction on where a value can be stored for override purposes (only axis is fixed).

### Alignment with Phase 03

- Reuse existing locale fallback chain from Phase 03 (`fallback_locale_code`).
- If channel fallback exists, integrate it the same way (e.g. try requested channel, then fallback channel) when resolving. If not yet implemented, add it in line with locale fallback so inheritance + locale + channel work in unison.

## 2. Product UI: Always Show First Available Fallback with Inheritance Indicator

### Requirement

- In the product edit UI, **always show the first available fallback value** (from parent, locale fallback, or channel fallback) in the main editing area — not a blank when a fallback exists.
- The value must be shown **together with an identifier** that it is inherited (e.g. “Inherited from …”, “From locale …”, “From parent …”).
- **If override is allowed** (non-axis attribute, and backend allows override at this level), the UI must provide a way to **override** (e.g. edit field that writes a value on the current product, clearing the inherited display for that scope).

### Current Wrong Behavior

- UI may show a blank for a scope when the value is only available via parent or locale/channel fallback.
- Inherited values may be shown only in a separate “Inherited” block instead of in-place with a clear indicator and option to override.

### Correct Behavior

- For every attribute field in the product edit form (per channel/locale context):
  - **Display:** Show the resolved value (first available from self, then parent + fallbacks). Never show empty when a fallback value exists.
  - **Indicator:** Clearly label when the value is inherited (e.g. “From parent PRD-00003”, “From locale en”, “From channel Default”). Use existing “Inherited from …” pattern but ensure it applies to locale/channel fallback as well as hierarchy.
  - **Override:** If the attribute is overridable at this level (non-axis, and API allows write), provide an explicit control to set/override the value on the current product. Once overridden, the displayed value is “own” and the indicator reflects that.

So: One consistent behavior — show first available value everywhere, mark source, and allow override where permitted.

## 3. Locale Completeness Reset & Per-Variant, Interactive Completeness

### Requirement

**3.1 Completeness “reset” in the locale fallback chain**

- When configuring locales and fallback chains, the user must be able to specify **where in the chain the completion score is “reset”**.
- **Example:** Fallback chain `en` → `de` → `de-DE` (and e.g. `en` → `en-US`). If `en` is fully filled, today the system may consider `de` and `de-DE` complete because they inherit values from `en`. That is wrong when the intent is “German locales must have their own German content to count as complete.”
- **Desired behavior:** Configure a “completeness boundary” between `en` and `de`. Then:
  - **Display / resolution:** `de` (and `de-DE`) still **inherit** and **show** values from `en` (first available value).
  - **Completeness for `de` (and descendants in the chain):** Values that exist **only** via fallback from `en` do **not** count toward completeness. Only values stored **in** `de` (or in a locale that falls back to `de` without crossing another reset) count. So filling `en` does not make `de` or `de-DE` complete; the user must add German content in `de` / `de-DE` to improve their completeness score.

**3.2 Schema / configuration**

- Add a flag on the **locale** (e.g. `completeness_resets_after_fallback` or `completeness_boundary`). When **true** for locale `L`:
  - When calculating completeness **for locale `L`**, a required attribute is counted as “present” **only** if there is a value stored with `locale_code = L` (or with a locale that falls back to `L` without crossing another such boundary). Values found only via `L`’s fallback (e.g. `fallback_locale_code`) do **not** count.
- So the “break” is between `L` and its fallback: completeness for `L` does not count inherited-from-fallback values.

**3.3 Completeness calculated per variant, per locale; then aggregated**

- Completeness scores must be calculated **per variant** and **per locale** (and per channel), using the reset rule above.
- These per-variant, per-locale (and per-channel) scores are then **aggregated** (e.g. summed or averaged) to provide an **overall completion score** for the product (or for the view: e.g. per product model, or per channel).
- Exact aggregation (e.g. average vs weighted sum) can be defined in Phase 11 or here; the requirement is that the building blocks are per-variant and per-locale with reset, and that an overall score is derivable.

**3.4 Interactive completeness**

- The completeness score in the UI must be **interactive** and **able to identify what part of the product is not complete**:
  - e.g. which **variant(s)**,
  - which **locale(s)** (and channel if applicable),
  - and which **attributes** are missing or only filled via a fallback that doesn’t count (when reset is used).
- So: click or hover on a completeness indicator to see a breakdown (by variant, locale, channel, missing attributes) and, where possible, deep-link or scroll to the relevant field.

## 4. Summary of Changes by Area

| Area | Change |
|------|--------|
| **Phase 09 (Values)** | Allow override: any level below the attribute’s owner level may write a value (except axis). Read: first available value = hierarchy + locale (and channel) fallback in unison. Annotate source (own / inherited / fallback). |
| **Phase 03 (Locales)** | Add locale flag: e.g. `completeness_resets_after_fallback`. When true, completeness for that locale does not count values found only via its fallback. |
| **Phase 11 (Completeness)** | Completeness uses “first available” value for display/API, but for “present” in completeness evaluation, apply locale reset: if locale has reset, only count values stored in that locale (or downstream in its chain without crossing another reset). Calculate per variant, per locale (and per channel); then aggregate for overall score. Expose missing-attribution breakdown (variant, locale, channel, attributes). |
| **Phase 12 (Product UI)** | Always show first available fallback value in the form; show inheritance/fallback indicator; allow override where permitted. Completeness indicator interactive: identify incomplete parts (variant, locale, channel, attributes). |

---

## 5. Exit Criteria

- [ ] **Override:** API and services allow writing non-axis attributes at any level below the owner level; read path returns first available value (hierarchy + locale + channel fallback) with source annotation.
- [ ] **UI:** Product edit always shows first available value with clear inherited/fallback indicator; override control where allowed.
- [ ] **Locales:** Locale configuration supports “completeness reset” (boundary) so that completeness for a locale does not count values only from its fallback.
- [ ] **Completeness:** Per-variant, per-locale (per-channel) completeness with reset; aggregated overall score; interactive UI that identifies what is incomplete (variant, locale, channel, attributes).
