## Objective

Enable **machine-to-machine** access to the PIM API via OAuth 2.0 **Client Credentials** flow. This is separate from user login (Phase 01): a separate Entra ID app registration (or a second application/secret under the same tenant) is used to obtain access tokens that can call the API on behalf of a service, not a user.

Required before external consumers (e.g., Akeneo import script, sync jobs, headless storefronts) can authenticate to the REST API without a browser session.

---

## 1. Core Concepts

- **User auth (Phase 01):** Users sign in with Entra ID; session cookie; requests are associated with a user identity.
- **Service auth (this phase):** A client (script, service, another app) sends `client_id` + `client_secret` to Entra ID, receives an access token, and sends that token in `Authorization: Bearer <token>` to the PIM API. No user session; the request is associated with an application/client identity.
- **Entra ID:** A second app registration (or "Application" type registration) is used for the machine client. That registration has a client secret (or certificate). It is configured to allow the Client Credentials flow and to request access to the PIM API (or a scope that the PIM API validates).
- **PIM API:** A middleware or guard validates the Bearer token (e.g., via Entra ID token validation or introspect), ensures it is for the expected audience/scope, and treats the request as an authenticated "service" call. Optionally map to a synthetic user or role (e.g., "system" or "api-client") for audit and RBAC.

---

## 2. Entra ID Configuration

- **App registration (service client):** Create an application registration in the same Entra tenant (or document use of an existing one). Application type: suitable for client credentials (no redirect URIs required for this flow).
- **Client secret (or certificate):** Generate a secret (or upload a certificate) for the app. The client uses this to authenticate to Entra ID.
- **API permissions:** The app registration needs permission to access the PIM API. Options:
  - If the PIM API is exposed as an Entra-protected API (another app registration with App ID URI and scopes), grant the service client that API permission (e.g., `api://pim-api/.default` or a dedicated scope like `api://pim-api/access`).
  - If the PIM API validates tokens by audience only (e.g., accepts any token with audience = PIM backend app ID), grant the service client permission to that backend app (e.g., "Access PIM API").
- **Token endpoint:** Client uses the standard Microsoft identity platform token endpoint (e.g., `https://login.microsoftonline.com/<tenant-id>/oauth2/v2.0/token`) with `grant_type=client_credentials`, `client_id`, `client_secret`, and `scope` (e.g., `api://pim-api-id/.default`).

Document the exact steps (tenant ID, app ID, secret creation, scope) in a runbook or README so that future consumers (e.g., Akeneo import) can register their own client or use a shared service account.

---

## 3. PIM Backend Behavior

- **Token validation:** For requests that present `Authorization: Bearer <token>`:
  - Validate the token (signature, expiry, audience, issuer) using the same tenant and app configuration as user auth (or a dedicated validator for service tokens).
  - Ensure the token was issued for the Client Credentials flow (no user claim, or distinct `azp`/appid claim).
- **Identity for the request:** After validation, associate the request with a service identity. Options:
  - **Option A:** Create a synthetic "system" or "api-client" user (or use a fixed service account) and attach it to the request (e.g., `req.serviceClientId = appId` and `req.user = systemUser` for audit).
  - **Option B:** Store the client app ID and optional custom claims; use that for audit (e.g., "created by client abc-123") and skip user-based RBAC for that request.
- **RBAC:** Decide whether service clients have a fixed role (e.g., full API access) or whether you introduce "API keys" / "service accounts" in the PIM DB that map client_id to a role. MVP can be: all validated service tokens get a single high-privilege role for API access; tighten in Week 4 (RBAC enforcement).
- **Audit:** Log all API requests authenticated via client credentials with the client app ID (and optional user/system identifier) so that imports and integrations are traceable.

---

## 4. API Contract

- **No change to REST response shapes.** Same endpoints, same responses. Only the way the caller authenticates changes (Bearer token instead of session cookie).
- **Optional:** A dedicated health or "who am I" endpoint (e.g., `GET /api/v1/whoami` or `GET /api/v1/auth/me`) that returns current identity: for user auth returns user info; for Bearer service token returns client app ID and/or "service" role. Useful for debugging and for consumers to verify they are authenticated.

---

## 5. Security Considerations

- **Secrets:** Client secrets must not be committed to source control. Document that consumers use environment variables or a secret manager. The PIM backend never stores client secrets; it only validates tokens issued by Entra ID.
- **Scope:** Use a dedicated scope (e.g., `api://pim-api/access`) so that only intended clients can request access to the PIM API.
- **Rotation:** Document how to rotate the client secret in Entra ID and update consumers. Short-lived tokens (e.g., 1 hour) limit exposure if a token leaks.

---

## 6. Exit Criteria

- [ ] A second Entra ID app registration (or equivalent) is created and documented for client credentials.
- [ ] A client can obtain an access token using client_id + client_secret and the token endpoint.
- [ ] The PIM API accepts `Authorization: Bearer <token>`, validates the token, and treats the request as an authenticated service call.
- [ ] Audit logs record service-authenticated requests with client identity.
- [ ] Documentation or runbook exists for registering a new service client and calling the API (e.g., for Akeneo import or other integrations).
