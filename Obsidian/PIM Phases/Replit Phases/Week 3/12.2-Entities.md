## Objective

Build **entities** (reference entities / metaobjects): standalone structured records with their own attribute definitions, same pattern as products but without variant hierarchy. Entities use the same EAV attribute value system, localizable and channelizable scoping, and audit history. They are referenced from products via the `entityReference` attribute type (product stores the entity code).

Use cases: brand profiles, material specs, care instructions, compliance records, designer bios — any structured data that products link to but that is managed as its own record.

---

## 1. Core Concepts

- An **entity type** (like a family for products) defines which attributes an entity record has. Each entity type has a code, labels, and a set of attributes with optional requirements per channel/locale.
- An **entity** is a single record of a given entity type. It has a unique code (camelCase), belongs to one entity type, and has attribute values stored in the shared `attribute_values` table with `entity_type = 'entity'` and `entity_id` = entity `code`.
- Entities have **no variant hierarchy** — no parent/child, no axes. Flat list of records per type.
- Attribute values use the same scoping: global, localizable, channelizable, or both. Reuse the existing `attribute_values` table and scoping logic from Phase 03/09.
- **Audit history** is wired for all entity and entity-type create/update/delete operations.

## 2. Database Schema

### Table: `entity_types`

Defines the "schema" for an entity — which attributes it has and which are required.

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `code` | `VARCHAR(100)` | PK | camelCase identifier (e.g., `brand`, `materialSpec`) |
| `labels` | `JSONB` | NOT NULL, DEFAULT '{}' | Display name per locale |
| `created_at` | `TIMESTAMPTZ` | NOT NULL, DEFAULT now() | |
| `updated_at` | `TIMESTAMPTZ` | NOT NULL, DEFAULT now() | |

### Table: `entity_type_attributes`

Junction: which attributes belong to which entity type (like `family_attributes`).

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `entity_type_code` | `VARCHAR(100)` | FK → entity_types.code, ON DELETE CASCADE | |
| `attribute_code` | `VARCHAR(255)` | FK → attributes.code, ON DELETE CASCADE | |
| `sort_order` | `INTEGER` | NOT NULL, DEFAULT 0 | Display order |
| `created_at` | `TIMESTAMPTZ` | NOT NULL, DEFAULT now() | |
| PRIMARY KEY | | `(entity_type_code, attribute_code)` | |

### Table: `entity_type_requirements`

Optional: which attributes are required per channel/locale (for completeness if needed later). Same shape as `family_attribute_requirements` but for entity types. Can be deferred to a later phase if entity completeness is not in MVP scope.

### Table: `entities`

The entity records themselves.

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `code` | `VARCHAR(255)` | PK | camelCase identifier (e.g., `leatherman`, `420hc`) |
| `entity_type_code` | `VARCHAR(100)` | NOT NULL, FK → entity_types.code | Which type this entity is |
| `created_at` | `TIMESTAMPTZ` | NOT NULL, DEFAULT now() | |
| `updated_at` | `TIMESTAMPTZ` | NOT NULL, DEFAULT now() | |

**Indexes:** `entity_type_code`, `updated_at`.

### Attribute Values

Reuse the existing `attribute_values` table with:

- `entity_type = 'entity'`
- `entity_id` = entity `code`

All existing columns (channel_code, locale_code, value_*, etc.) work as for products. Service layer restricts reads/writes to attributes that belong to the entity's type. *(Implementation confirmed: shared table uses entity_type/entity_id; no separate resource_type column.)*

## 3. API Endpoints

### Entity Types

| Method | Endpoint | Description |
|--------|----------|-------------|
| GET | `/api/v1/entity-types` | List all entity types (cursor pagination: `limit`, `cursor`; optional `search`). May return `attributeCount` per type. |
| GET | `/api/v1/entity-types/:code` | Get one entity type with its attributes and optional requirements |
| POST | `/api/v1/entity-types` | Create entity type (code, labels). Code must be camelCase. |
| PATCH | `/api/v1/entity-types/:code` | Update labels |
| DELETE | `/api/v1/entity-types/:code` | Delete type; cascade to entities and their values or block if entities exist |
| POST | `/api/v1/entity-types/:code/attributes` | Add attributes to type. Body: `attributes` array with `attributeCode` and optional `sortOrder`. Auto-increment from max if omitted. Validates attribute exists and is not already assigned. |
| DELETE | `/api/v1/entity-types/:code/attributes/:attributeCode` | Remove attribute from type (does not delete attribute definition) |

### Entities

| Method | Endpoint | Description |
|--------|----------|-------------|
| GET | `/api/v1/entities` | List entities. Query: `entityTypeCode`, `search`, cursor pagination. |
| GET | `/api/v1/entities/:code` | Get one entity with its type and attribute values. Optional `channel` and `locale` query params for scoped value retrieval with locale fallback. |
| POST | `/api/v1/entities` | Create entity. Body: `entityTypeCode`, `code` (camelCase). |
| PATCH | `/api/v1/entities/:code` | Update (e.g. code change if allowed; typically code is immutable) |
| DELETE | `/api/v1/entities/:code` | Delete entity and its attribute values (cascade) |
| GET | `/api/v1/entities/:code/values` | Get attribute values. Query: `channel`, `locale`. Locale fallback resolution with source annotation (`own` / `fallback` with `fromLocale`). Same response shape as product values. |
| PUT | `/api/v1/entities/:code/values` | Set attribute values. Body: `values` array (e.g. up to 500). Same request/response pattern as products. Validate attribute belongs to entity type and type-specific validation. |

## 4. Validation & Business Rules

- **Entity type code** and **entity code**: camelCase (reuse shared schema). Immutable after create.
- **Entity create**: `entityTypeCode` must exist. `code` must be unique across all entities.
- **Values**: Only attributes that belong to the entity's type can be written. Same type-specific validation (required, regex, min/max, etc.) as products. No variant inheritance — all values are owned by the entity.
- **Delete entity type**: Return 400 if any entities of that type exist (must delete entities first). No cascade delete of entity type.

## 8. Getting Started — Implementation Order (Reference)

Use this order when building or extending Phase 12.2 in the codebase.

1. **Schema & migrations**
   - Add tables: `entity_types`, `entity_type_attributes`, `entities`. (Defer `entity_type_requirements` if entity completeness is out of scope.)
   - Confirm `attribute_values` supports `entity_type = 'entity'` and `entity_id` = entity `code` (or adjust column names to match existing product convention, e.g. `resource_type` / `resource_id`).

2. **Entity types API**
   - CRUD for entity types: list, get, create, patch, delete.
   - Endpoints to add/remove attributes on a type (`POST .../attributes`, `DELETE .../attributes/:attributeCode`), with sort_order and auto-increment where applicable (per Phase 09.5).

3. **Entities API**
   - CRUD for entities: list (filter by `entityTypeCode`, search), get, create, patch, delete.
   - Reuse camelCase validation for entity type code and entity code.

4. **Entity values service & API**
   - Reuse the same value resolution pattern as products but without hierarchy: for each attribute, single record, with locale (and channel) fallback. Source is "own" for direct values or "fallback" with `fromLocale` when resolved via locale fallback chain.
   - `GET /api/v1/entities/:code/values` and `PUT /api/v1/entities/:code/values`; same request/response shape as product values. Validate that each attribute belongs to the entity's type; apply type-specific validation (reuse product validation helpers).

5. **Audit**
   - Wire audit logging for entity type and entity CRUD, and for entity value create/update/delete (same pattern as product values).

6. **Integration**
   - Ensure product value validation for `entityReference` attribute type can optionally validate that the stored code exists in `entities` (or leave to Phase 12.5 UI). No DB FK required.

After 12.2 is done, Phase 12.5 (Entity Management UI) can add list/create/edit screens and entity picker for product `entityReference` fields.
