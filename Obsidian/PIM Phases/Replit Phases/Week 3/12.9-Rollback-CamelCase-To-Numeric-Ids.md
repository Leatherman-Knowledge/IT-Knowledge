# Phase 12.9: Roll Back camelCase Codes to Auto-Incrementing Numeric IDs

**Execution Plan Item:** Follow-on to Phase 9.5 (camelCase enforcement) and structure/entity phases  
**Replit Phase:** 12.9 (Week 3)  
**Dependencies:** All phases that introduced or use camelCase codes for the entities listed below (Tasks 2, 3, 4, 4.5, 5, 12.2, 12.3; Phase 9.5)  
**Depended on by:** Future imports, API consumers; no downstream phase is blocked if this is deferred

---

## Objective

Roll back the use of **camelCase codes** as the primary identifier for most structure and entity types. Replace them with **auto-incrementing numeric IDs** in the database and **do not show codes in the UI**. Users identify items by labels (and optional list position); systems use numeric `id` in URLs and APIs.

**Scope:**

- **Database:** For each affected entity, the primary key becomes a numeric `id` (e.g. `SERIAL` / `GENERATED BY DEFAULT AS IDENTITY`). The `code` column is removed or deprecated; no new code values are generated or validated.
- **API:** Routes and request/response bodies use `id` (e.g. `GET /api/v1/attributes/:id`, `POST` bodies with `groupId` instead of `groupCode`). Query params that today use `*Code` use `*Id` or equivalent.
- **UI:** No code field or code column is shown. List and detail views use label (and optionally sort order); create flows do not ask for a code. Navigation and links use numeric `id`.

---

## Exceptions (Do Not Change)

The following **do not** switch to numeric IDs and are left as specified in earlier phases:

| Area | Current Behavior | Phase 12.9 Action |
|------|------------------|-------------------|
| **Top Models** | Identifiers use codes provided initially (e.g. from import), then auto-incrementing numbers for new products (Phase 9.6). | No change. Keep existing identifier scheme (initial codes + incrementing numbers). |
| **Middle models** | Product models with a parent use `{parent_identifier}_{incrementing_integer}` (e.g. `10_1`, `10_2`). | Leave as-is. No change. |
| **Variants** | User-provided SKU as product identifier (letters, numbers, or mixed). | No change. Variants remain user-provided SKUs. |
| **Locales** | BCP 47 code as PK (e.g. `en-US`). Not camelCase. | No change. |
| **Currencies** | ISO 4217 code as PK (e.g. `USD`). | No change. |
| **Roles** | String code PK (RBAC). | No change. |

Products (Top Model / Model / Variant), locale/currency tables, and roles are **out of scope** for this phase.

## Migration Order and Dependencies

Recommended order to avoid FK and reference cycles:

1. **Attribute groups** â€” no dependency on other code-based entities.
2. **Attributes** â€” depends on attribute groups (group_id).
3. **Attribute options** â€” depends on attributes.
4. **Families** â€” depends on attributes (label/image attribute id).
5. **Variant definitions** â€” depends on families.
6. **Category trees** â€” independent.
7. **Categories** â€” depends on category trees; update parent_id and path.
8. **Channels** â€” depends on category trees (category_tree_id).
9. **Entity types** â€” independent.
10. **Entities** â€” depends on entity types.
11. **Asset types** â€” independent.
12. **Assets** â€” depends on asset types.

Within each step:

- Add `id` column, backfill if needed (e.g. from existing sequence or generated values).
- Add new FK columns or update existing ones to reference `id`.
- Migrate data (map old code â†’ id in dependent tables).
- Update API and UI to use id only; remove code from request/response and screens.
- Drop `code` column (or mark deprecated and stop writing).

**Cross-cutting:** `attribute_values` references both attributes and entity_id. After attributes use `attribute_id`, update `attribute_values.attribute_code` â†’ `attribute_id`. For `entity_id`, the column may stay as string (storing string representation of numeric id for products/entities/assets) or become typed per entity_type; products already use numeric-like identifiers per Phase 9.6.

## API and UI Summary

- **Routes:** Use numeric `id` in path: `/attributes/1`, `/families/2`, `/entity-types/3`, etc.
- **Query params:** Prefer `*Id` (e.g. `groupId`, `familyId`, `entityTypeId`) instead of `*Code`.
- **Request bodies:** No `code` or `*Code` for create/update; use `id` where a reference is needed.
- **Response bodies:** Include `id`; omit `code` (or deprecate and remove).
- **UI:** No code input or code column anywhere for the 12 entities above. Lists and detail pages show label (and optionally sort order / count). Create dialogs: labels only (and type/group/etc. by id where applicable).

## Notes

- **Backward compatibility:** If existing integrations rely on codes, consider an optional read-only `code` column (populated from legacy data) or a temporary compatibility layer that maps id â†’ code for a transition period. This phase does not require it; product as specified is id-only.
- **Import/export:** Future imports (e.g. Akeneo) should map external codes to internal ids or create new rows with generated ids; code is no longer the primary key.
- **Reference types:** Attributes of type `entityReference`, `assetSingle`, `assetList`, `productReference` store the referenced entity's identifier. After this phase, that value is the target's numeric id (or string representation) for entities/assets; products already use their own identifier scheme (Top Model / Model / Variant) and are unchanged.
