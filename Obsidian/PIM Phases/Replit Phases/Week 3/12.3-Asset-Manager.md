## Objective

Build the **Asset Manager**: asset types (families) and asset records with attributes, using the same attribute infrastructure as products and entities. Products link to assets via existing `assetSingle` and `assetList` attribute types (product stores asset code(s)). This phase covers schema, API, and audit — not the full asset picker UI (that is Phase 12.6).

Use cases: product photos, lifestyle images, technical drawings, videos, documents — each with metadata (alt text, photographer, usage rights, etc.) via attributes.

---

## 1. Core Concepts

- An **asset type** (like a family for products) defines which attributes an asset has (e.g., "Product Photo" has alt text, photographer, usage rights). Each asset type has a code, labels, and a set of attributes with optional requirements.
- An **asset** is a single record of a given asset type. It has a unique code (camelCase), belongs to one asset type, and has attribute values — including at least one that holds the actual file/media (e.g., a `mediaFile` attribute). Other attributes can be text, date, single select, etc.
- **Linking to products:** Products already have `assetSingle` and `assetList` attributes (Phase 02). The value stored is the asset code. No new link table — product attribute values reference assets by code. This phase ensures assets exist and are manageable so those product attributes can reference them.
- Attribute values use the same scoping (global, localizable, channelizable). Reuse `attribute_values` with `entity_type = 'asset'`, `entity_id` = asset code.
- **Audit history** wired for all asset type and asset create/update/delete operations.

## 2. Database Schema

### Table: `asset_types`

Defines the "schema" for an asset — which attributes it has.

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `code` | `VARCHAR(100)` | PK | camelCase identifier (e.g., `productPhoto`, `lifestyleImage`) |
| `labels` | `JSONB` | NOT NULL, DEFAULT '{}' | Display name per locale |
| `created_at` | `TIMESTAMPTZ` | NOT NULL, DEFAULT now() | |
| `updated_at` | `TIMESTAMPTZ` | NOT NULL, DEFAULT now() | |

### Table: `asset_type_attributes`

Junction: which attributes belong to which asset type.

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `asset_type_code` | `VARCHAR(100)` | FK → asset_types.code, ON DELETE CASCADE | |
| `attribute_code` | `VARCHAR(255)` | FK → attributes.code, ON DELETE CASCADE | |
| `sort_order` | `INTEGER` | NOT NULL, DEFAULT 0 | Display order |
| `created_at` | `TIMESTAMPTZ` | NOT NULL, DEFAULT now() | |
| **PRIMARY KEY** | | `(asset_type_code, attribute_code)` | |

**Indexes:** `idx_ata_asset_type` on `asset_type_code`.

### Table: `assets`

The asset records.

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `code` | `VARCHAR(255)` | PK | camelCase identifier (e.g., `wingmanHeroFront`) |
| `asset_type_code` | `VARCHAR(100)` | NOT NULL, FK → asset_types.code | Which type this asset is |
| `created_at` | `TIMESTAMPTZ` | NOT NULL, DEFAULT now() | |
| `updated_at` | `TIMESTAMPTZ` | NOT NULL, DEFAULT now() | |

**Indexes:** `idx_assets_type` on `asset_type_code`, `idx_assets_updated` on `updated_at`.

### Attribute Values

Reuse `attribute_values` with:

- `entity_type = 'asset'`
- `entity_id` = asset `code`

*(Implementation aligns with entities and products: shared table uses `entity_type` / `entity_id`.)* No new columns or tables needed — all existing value columns (`value_text`, `value_number`, `value_boolean`, `value_date`, `value_json`, `value_media`) and scoping columns (`channel_code`, `locale_code`) work as-is. At least one attribute on an asset type should typically be `mediaFile` (or similar) so the asset has actual content; the rest are metadata.

## 3. API Endpoints

### Asset Types

| Method | Endpoint | Description |
|--------|----------|-------------|
| GET | `/api/v1/asset-types` | List all asset types with cursor pagination (`limit`, `cursor`) and optional `search` query. Returns `attributeCount` per type. |
| GET | `/api/v1/asset-types/:code` | Get one asset type with its assigned attributes array. |
| POST | `/api/v1/asset-types` | Create asset type. Body: `code` (camelCase), `labels`. Validates uniqueness. |
| PATCH | `/api/v1/asset-types/:code` | Update labels. Code is immutable. |
| DELETE | `/api/v1/asset-types/:code` | Delete type. Returns 400 if any assets of this type exist (must delete assets first). |
| POST | `/api/v1/asset-types/:code/attributes` | Add attributes to type. Body: `attributes` array with `attributeCode` and optional `sortOrder`. Auto-increments sort order from max if omitted. Validates attribute exists and is not already assigned. |
| DELETE | `/api/v1/asset-types/:code/attributes/:attributeCode` | Remove attribute from type (does not delete the attribute definition itself). |

### Assets

| Method | Endpoint | Description |
|--------|----------|-------------|
| GET | `/api/v1/assets` | List assets with `assetTypeCode`, `search`, `limit`, `cursor` query params. Cursor-based pagination. |
| GET | `/api/v1/assets/:code` | Get one asset with its asset type definition and attribute values. Optional `channelCode` and `localeCode` query params for scoped value retrieval with locale fallback. |
| POST | `/api/v1/assets` | Create asset. Body: `code` (camelCase), `assetTypeCode`. Validates type exists and code is unique. |
| PATCH | `/api/v1/assets/:code` | Update endpoint. `code` and `assetTypeCode` are immutable. |
| DELETE | `/api/v1/assets/:code` | Delete asset and cascade-delete all its attribute values. |
| GET | `/api/v1/assets/:code/values` | Get attribute values with optional `channelCode` and `localeCode` scoping. Includes locale fallback resolution with source annotation. |
| PUT | `/api/v1/assets/:code/values` | Set attribute values. Body: `values` array (up to 500). Validates each attribute belongs to the asset's type and passes type-specific validation (scoping, type coercion). |

## 4. Validation & Business Rules

- **Asset type code** and **asset code**: Validated against strict camelCase regex (`^[a-z][a-zA-Z0-9]*$`). Immutable after creation.
- **Asset create**: `assetTypeCode` must reference an existing asset type. `code` must be globally unique across all assets.
- **Values write**: Only attributes assigned to the asset's type can be written. Attribute scoping (localizable/channelizable) is enforced by the shared `setAttributeValue` service. Type-specific validation (number precision, boolean type, date parsing, mediaFile handling, etc.) applied per attribute type.
- **Delete asset type**: Returns 400 with count if any assets of that type exist — must delete assets first.
- **Delete asset**: Cascade-deletes all attribute values for the asset from `attribute_values` before removing the asset record.

---

## 5. Locale Fallback Resolution

Aligned with Phase 12.1 and 12.2 patterns. The asset values read path (`server/lib/asset-values.ts`) implements:

- **Own values**: Returned with `source: { type: "own" }`.
- **Fallback values**: When a localizable attribute has no value for the requested locale, the service walks the locale fallback chain (e.g., `de-DE` → `de` → `en`) and returns the first non-empty value with `source: { type: "fallback", fromLocale: "en" }`.
- **No variant hierarchy**: Assets are flat — no parent/child inheritance. Only locale fallback applies.
- Values are filtered by the asset type's attribute membership, so only relevant attributes are returned.

## 6. Audit History

All operations are audit-logged using the existing `auditHistory` table and `logAudit` / `logAuditBatch` functions.

| Resource Type | Actions Logged |
|---------------|----------------|
| `assetType` | create, update, delete, attribute add, attribute remove |
| `asset` | create, update, delete |
| `asset` (values) | Per-attribute value create/update via `logAuditBatch` with `attributeCode`, `channelCode`, `localeCode`, `oldValue`, `newValue` |

Validation of asset code existence when writing product values is deferred to Phase 12.6 (Asset Management UI).
