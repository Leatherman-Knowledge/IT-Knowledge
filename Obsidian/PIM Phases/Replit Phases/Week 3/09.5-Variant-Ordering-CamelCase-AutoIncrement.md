# Task 9.5: Variant Ordering, camelCase Enforcement & Auto-Incrementing Values

**Execution Plan Item:** #9 (from execution plan — distinct from Task 9 spec document)
**Dependencies:** Products (Task 8), Product Attribute Values (Task 9), Families (Task 4), Variant Definitions (Task 4.5)
**Depended on by:** Product UI (Task 12)

## Objective

Three cross-cutting concerns that need to be baked into the product layer now rather than retrofitted later. These are small individually but affect multiple surfaces — schema, validation, API behavior — so they're bundled together.

1. **Variant ordering** — Persistent display order for child products under a parent model.
2. **camelCase enforcement** — Consistent code format validation across all entity creation endpoints.
3. **Auto-incrementing sort orders** — When entities are added to ordered collections, default sort order values should auto-increment so explicit ordering is optional.

## 1. Variant Ordering

### Problem

The `products` table has no mechanism for controlling the display order of child products under a parent model. When the UI renders a variant hierarchy — e.g., the children of `PRD-00003` — there's no way to control whether "red" appears before "blue." Currently, order would default to whatever the database returns (insertion order or identifier alphabetical), which isn't reliable or user-controllable.

### Schema Change

Add a `sort_order` column to the `products` table:

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `sort_order` | `INTEGER` | NOT NULL, DEFAULT 0 | Display order among siblings. Lower values sort first. Only meaningful for child products (children of the same parent are sorted by this field). |

```sql
ALTER TABLE products ADD COLUMN sort_order INTEGER NOT NULL DEFAULT 0;
CREATE INDEX idx_products_parent_sort ON products(parent_identifier, sort_order);
```

### Behavior

**On child creation:**
When creating a child product under a parent, if `sortOrder` is not provided in the request body, auto-assign the next sort order value among the parent's existing children:

```
sortOrder = MAX(sort_order among siblings) + 1
```

If the parent has no existing children, default to `0`.

If `sortOrder` is explicitly provided, use it as-is. No gap-filling or reindexing — sort orders can have gaps (e.g., 0, 1, 5, 10). This keeps inserts fast and avoids updating sibling rows.

**On simple product and root model creation:**
`sort_order` defaults to `0`. It has no ordering significance for root-level products (they're sorted by other criteria in list views — see below).

**On product listing:**
- `GET /api/v1/products/:identifier/children` and `GET /api/v1/products/:identifier/tree` should order children by `sort_order ASC, identifier ASC` (identifier as tiebreaker).
- `GET /api/v1/products?parentIdentifier=X` should also respect this order.
- Root-level product listings (`GET /api/v1/products?root=true`) sort by `created_at DESC` by default (newest first), not by sort_order.

### Reordering Endpoint

**`PATCH /api/v1/products/:identifier/reorder`** — Reorder children of a product model.

Request body:
```json
{
  "orderedIdentifiers": [
    "PRD-00003-blue",
    "PRD-00003-red"
  ]
}
```

The array must contain the identifiers of **all** direct children of the product. The position in the array determines the new `sort_order` (index 0 → sort_order 0, index 1 → sort_order 1, etc.).

**Validation:**
- The target product must be a `model` type (not `simple` or `variant`). Return 400 otherwise.
- The `orderedIdentifiers` array must contain exactly the set of current direct children — no missing, no extras, no duplicates. Return 400 with a descriptive error if the list doesn't match.
- All updates happen in a single transaction.

**Response:** Returns the updated children list in the new order.

### PATCH Products Update

Add `sortOrder` as an optional field on `PATCH /api/v1/products/:identifier`. This allows reordering a single product without a full sibling reorder — useful for "move to top" or "move to bottom" operations.

### Audit

Log reorder operations as:

| Resource Type | Action | What's Logged |
|---------------|--------|---------------|
| `product` | reorder | `resourceId` = parent identifier, old order (array of identifiers), new order (array of identifiers) |

## 2. camelCase Enforcement

### Problem

Entity codes throughout the PIM should follow a consistent format. Task 4.5 introduced a `camelCaseCodeSchema` regex for variant definition codes, but the same convention needs to be enforced (or confirmed as enforced) across all entity types.

### Convention

All `code` fields follow **strict camelCase**:
- Starts with a lowercase letter
- Contains only letters (a-z, A-Z) and digits (0-9)
- No underscores, hyphens, spaces, or special characters
- First character is always lowercase

Regex: `^[a-z][a-zA-Z0-9]*$`

### Entities Requiring Enforcement

Audit each entity's creation endpoint and confirm the code field is validated against the camelCase regex. If not, add validation.

| Entity | Code Field | Create Endpoint | Status |
|--------|-----------|-----------------|--------|
| Attributes | `code` | `POST /api/v1/attributes` | Verify |
| Attribute Groups | `code` | `POST /api/v1/attribute-groups` | Verify |
| Families | `code` | `POST /api/v1/families` | Verify |
| Variant Definitions | `code` | `POST /api/v1/families/:code/variant-definitions` | Already enforced (Task 4.5) |
| Channels | `code` | `POST /api/v1/channels` | Verify |
| Categories | `code` | `POST /api/v1/categories` | Verify |
| Locales | `code` | `POST /api/v1/locales` | **Exception** — locale codes follow the `ll_CC` format (e.g., `en_US`, `fr_FR`), not camelCase. Do not apply camelCase validation to locale codes. |
| Attribute Options | `code` | `POST /api/v1/attributes/:code/options` | Verify |

### Validation Behavior

- Apply validation on **create** only. Codes are immutable after creation (already enforced across all entities), so update endpoints don't need this check.
- Return 400 with: `"Code must be camelCase (start with a lowercase letter, contain only letters and digits). Got: '...'"`
- If existing data has codes that violate camelCase (e.g., codes with underscores from seed data), those remain as-is. This is a forward-looking validation — only new entities must comply.

### Shared Validation Schema

Ensure `shared/validation.ts` exports a reusable schema:

```typescript
const camelCaseCodeSchema = z
  .string()
  .min(1)
  .max(100)
  .regex(/^[a-z][a-zA-Z0-9]*$/, "Code must be camelCase: start with a lowercase letter, letters and digits only");
```

All entity creation schemas should reference this shared schema for their `code` field (except locales).

## 3. Auto-Incrementing Sort Orders

### Problem

Multiple entities have `sort_order` fields used for display ordering: `family_attributes.sort_order`, `family_variant_axes.sort_order`, `family_variant_attribute_levels` (implicitly via display), attribute options, attribute groups, etc. When adding items to these ordered collections, the caller currently must explicitly provide a `sort_order` value. If omitted, items default to `0`, causing collisions and unpredictable ordering.

### Behavior

When an entity is added to an ordered collection and `sortOrder` is not provided in the request body, auto-assign the next value:

```
sortOrder = MAX(sort_order in collection) + 1
```

If the collection is empty, default to `0`.

### Affected Collections

| Collection | Sort Order Column | "Collection" Scope | When Auto-Increment Applies |
|-----------|-------------------|-------------------|---------------------------|
| Family attributes | `family_attributes.sort_order` | All attributes within a family | Adding an attribute to a family |
| Variant axes | `family_variant_axes.sort_order` | Axes at a given level within a definition | Adding axes (only when individual axis `sortOrder` not specified) |
| Attribute options | `attribute_options.sort_order` | Options within an attribute | Adding an option to an attribute |
| Attribute groups | `attribute_groups.sort_order` | All attribute groups globally | Creating an attribute group |
| Variant definitions | `family_variant_definitions.sort_order` | Definitions within a family | Creating a variant definition |
| Child products | `products.sort_order` | Children of a parent product | Creating a child product (covered in Section 1) |

### Important: Explicit Values Always Win

If the caller provides a `sortOrder` value, use it. Auto-increment only applies when the field is omitted or `null` in the request. This ensures bulk operations and data imports can set exact positions without interference.

### No Reindexing

Do not compact or reindex sort orders when items are deleted. Gaps are fine. If items at positions 0, 1, 2 exist and item 1 is deleted, the remaining items stay at 0 and 2. Reindexing would require updating multiple rows and locking — not worth it for a display order field.

## 4. Business Rules

1. **Variant sort order is scoped to siblings.** Each product's `sort_order` only has meaning relative to other children of the same parent. Changing a product's sort order does not affect products under a different parent.

2. **Sort order is mutable.** Unlike identifiers, sort orders can be changed at any time via PATCH or the reorder endpoint.

3. **camelCase is enforced on create, not retroactively.** Existing codes that violate camelCase continue to work. Only new entity creation is validated.

4. **Locale codes are exempt from camelCase.** They follow the IETF/CLDR `ll_CC` convention (e.g., `en_US`), which includes underscores and uppercase characters.

5. **Auto-increment sort order is a default, not a requirement.** Callers can always provide explicit sort orders. Auto-increment is a convenience for the common case where ordering doesn't matter at creation time.

6. **Reorder requires the complete child list.** Partial reordering (submitting only some children) is not supported — it's too easy to accidentally lose track of positions. The full list must be provided.

| Method | Purpose |
|--------|---------|
| `reorderProductChildren(parentIdentifier, orderedIdentifiers)` | Bulk update sort_order for all children of a parent in a single transaction. |
