# Task 8: Products & Product Models — Schema & Core CRUD

**Execution Plan Item:** #8
**Dependencies:** Families (Task 4), Variant Definitions (Task 4.5), Categories (Task 5), Attribute Values table (Task 3)
**Depended on by:** Everything product-related — attribute values on products, completeness, product UI

---

## Objective

Build the core product entity — the central record in the PIM. Products reference a family (which defines their attribute set), can optionally use a variant definition (which defines their hierarchy and axis structure), and can be organized into parent-child hierarchies for variant products.

This task covers:
- The `products` table with support for three product structures (simple, model with variants, model with two levels of variants)
- **Auto-generated sequential identifiers** for root products, with child identifiers derived from their root + axis values
- Parent-child relationships for variant hierarchies
- Integration with families and variant definitions
- Axis value tracking on variants (which option values define this variant's branch)
- REST API endpoints for full product CRUD
- Audit history wiring for all product operations

---

## 1. Product Structures

The system supports three product structures, all stored in a single `products` table with a `type` discriminator:

### Simple Product
A standalone product with no variants. All attribute values live directly on the product.

```
[Simple Product: PRD-00001]
  family: knife
  type: simple
  parent: null
  All attributes at level 0
```

### Product Model with One Axis Level
A parent product model with variant children. The model holds shared (level 0) attribute values. Each variant holds its own level-1 attribute values and is defined by its axis value(s).

```
[Product Model: PRD-00002]              ← type: model, level: 0
  family: multitool
  variant definition: colorOnly
  ├── [Variant: PRD-00002-black]        ← type: variant, level: 1, axis: { handleColor: "black" }
  ├── [Variant: PRD-00002-silver]       ← type: variant, level: 1, axis: { handleColor: "silver" }
  └── [Variant: PRD-00002-red]          ← type: variant, level: 1, axis: { handleColor: "red" }
```

### Product Model with Two Axis Levels
A root product model with intermediate sub-models (level 1) and leaf variants (level 2). Attributes cascade: root → sub-model → variant.

```
[Product Model: PRD-00003]                          ← type: model, level: 0
  family: knife
  variant definition: colorThenBlade
  ├── [Sub-Model: PRD-00003-red]                    ← type: model, level: 1, axis: { handleColor: "red" }
  │   ├── [Variant: PRD-00003-red-plain]            ← type: variant, level: 2, axis: { bladeType: "plain" }
  │   └── [Variant: PRD-00003-red-serrated]         ← type: variant, level: 2, axis: { bladeType: "serrated" }
  ├── [Sub-Model: PRD-00003-blue]                   ← type: model, level: 1, axis: { handleColor: "blue" }
  │   ├── [Variant: PRD-00003-blue-plain]           ← type: variant, level: 2, axis: { bladeType: "plain" }
  │   └── [Variant: PRD-00003-blue-serrated]        ← type: variant, level: 2, axis: { bladeType: "serrated" }
```

## 2. Database Schema

### Table: `products`

Single table for all product types — simple products, product models (root and sub-models), and leaf variants.

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `identifier` | `VARCHAR(255)` | PK | Auto-generated for root products (`PRD-00001`). Auto-derived for children (parent identifier + axis value codes). See Section 3. |
| `product_number` | `BIGINT` | GENERATED BY DEFAULT AS IDENTITY, NULL, UNIQUE | Sequential counter. Assigned to root products only (simple + root models). NULL for sub-models and variants. |
| `family_code` | `VARCHAR(100)` | NOT NULL, FK → families.code | Which family defines this product's attribute set |
| `variant_definition_code` | `VARCHAR(100)` | NULL | Which variant definition this hierarchy uses. Set on root models only. Sub-models and variants inherit from root. |
| `parent_identifier` | `VARCHAR(255)` | FK → products.identifier, NULL | Parent product. NULL for simple products and root models. |
| `type` | `VARCHAR(20)` | NOT NULL | `simple`, `model`, or `variant` |
| `level` | `INTEGER` | NOT NULL, DEFAULT 0 | 0 = root model or simple product, 1 = first-level child, 2 = second-level child, etc. |
| `axis_values` | `JSONB` | NULL | For children: the axis attribute value(s) that define this branch. e.g., `{ "handleColor": "red" }` |
| `status` | `VARCHAR(20)` | NOT NULL, DEFAULT 'draft' | `draft`, `active`, or `archived` |
| `created_at` | `TIMESTAMPTZ` | NOT NULL, DEFAULT now() | |
| `updated_at` | `TIMESTAMPTZ` | NOT NULL, DEFAULT now() | |

**Foreign key note:** `(family_code, variant_definition_code)` should reference `family_variant_definitions(family_code, code)` when `variant_definition_code` is not NULL. However, since this is a conditional FK (only applies to root models), enforce at the application layer rather than as a database constraint.

**Indexes:**

```sql
CREATE INDEX idx_products_family ON products(family_code);
CREATE INDEX idx_products_parent ON products(parent_identifier);
CREATE INDEX idx_products_type ON products(type);
CREATE INDEX idx_products_status ON products(status);
CREATE INDEX idx_products_family_status ON products(family_code, status);
CREATE INDEX idx_products_number ON products(product_number) WHERE product_number IS NOT NULL;
```

**Unique constraint on axis values within a parent:**

```sql
CREATE UNIQUE INDEX uq_products_parent_axes
ON products (parent_identifier, axis_values)
WHERE parent_identifier IS NOT NULL;
```

This prevents creating two variants under the same parent with the same axis value combination (e.g., two "red" variants under the same model).

## 3. Identifier Generation

Product identifiers are **system-generated**, not user-supplied. This keeps identifiers consistent, predictable, and collision-free.

### Root Products (Simple + Root Models)

Root product identifiers are generated from the `product_number` sequence:

```
Format:  PRD-{number padded to 5 digits}
Example: PRD-00001, PRD-00042, PRD-01337
```

The generation process:
1. Insert the product row. The `product_number` column auto-increments via `GENERATED BY DEFAULT AS IDENTITY`.
2. Format the identifier as `PRD-{product_number.toString().padStart(5, '0')}`.
3. Set the `identifier` column to the formatted value.

If the catalog grows beyond 99,999 products, the number simply extends: `PRD-100000`. The 5-digit padding is a display convention, not a hard limit.

### Children (Sub-Models + Variants)

Child identifiers are **derived from the parent identifier + the axis option codes** at that child's level, joined with hyphens:

```
Format:  {parent_identifier}-{axisValue1}-{axisValue2}-...
```

The axis values are the option codes from the axis attributes, joined in the order the axes are defined in the variant definition.

**Examples:**

| Product | Level | Axis Values | Generated Identifier |
|---------|-------|-------------|---------------------|
| Root model | 0 | — | `PRD-00003` |
| Sub-model (color: red) | 1 | `{ "handleColor": "red" }` | `PRD-00003-red` |
| Variant (blade: plain) | 2 | `{ "bladeType": "plain" }` | `PRD-00003-red-plain` |
| Sub-model (color: blue) | 1 | `{ "handleColor": "blue" }` | `PRD-00003-blue` |
| Variant (blade: serrated) | 2 | `{ "bladeType": "serrated" }` | `PRD-00003-blue-serrated` |

For a level with **multiple axis attributes** (cartesian product axes), the values are joined in the axis sort order:

```
Axes at level 1: [handleColor, sheathType]
Axis values: { "handleColor": "red", "sheathType": "leather" }
Identifier: PRD-00003-red-leather
```

### Identifier Rules

- **Identifiers are immutable.** Once generated, they never change.
- **Identifiers are globally unique** across all product types.
- **The `identifier` column is the PK** and is used in all foreign references (`attribute_values.entity_id`, `product_categories.product_identifier`, API URLs, etc.).
- **Users never supply an identifier.** The API does not accept `identifier` in the create body. It is always returned in the response.

### Implementation

```typescript
function generateRootIdentifier(productNumber: number): string {
  return `PRD-${productNumber.toString().padStart(5, '0')}`;
}

function generateChildIdentifier(
  parentIdentifier: string,
  axisValues: Record<string, string>,
  axisAttributeCodes: string[]  // in definition sort order
): string {
  const axisSuffix = axisAttributeCodes
    .map(code => axisValues[code])
    .join('-');
  return `${parentIdentifier}-${axisSuffix}`;
}
```

## 4. API Endpoints

### Products — CRUD

**`GET /api/v1/products`** — List products. Supports filters:
  - `?familyCode=knife` — filter by family
  - `?type=simple,model` — filter by product type (comma-separated)
  - `?status=draft,active` — filter by status
  - `?parentIdentifier=PRD-00003` — get direct children of a product model
  - `?root=true` — only root-level products (simple + root models, excludes children)
  - `?search=PRD-00003` — search by identifier or product number
  - Standard cursor-based pagination (`limit`, `cursor`)

Response includes: identifier, productNumber, familyCode, type, level, status, parentIdentifier, axisValues, createdAt, updatedAt.

**`GET /api/v1/products/:identifier`** — Get a single product with:
  - Its family code and variant definition code (resolved from root)
  - Its parent identifier (if a child)
  - Its direct children (if a model)
  - Its axis values
  - Its ancestor chain (for breadcrumb display)

Response structure:
```json
{
  "data": {
    "identifier": "PRD-00003-red-plain",
    "productNumber": null,
    "familyCode": "knife",
    "variantDefinitionCode": "colorThenBlade",
    "parentIdentifier": "PRD-00003-red",
    "type": "variant",
    "level": 2,
    "axisValues": { "bladeType": "plain" },
    "status": "draft",
    "ancestors": [
      { "identifier": "PRD-00003", "productNumber": 3, "level": 0, "axisValues": null },
      { "identifier": "PRD-00003-red", "productNumber": null, "level": 1, "axisValues": { "handleColor": "red" } }
    ],
    "children": [],
    "createdAt": "2026-02-20T10:00:00Z",
    "updatedAt": "2026-02-20T10:00:00Z"
  }
}
```

**`POST /api/v1/products`** — Create a product. The identifier is **always auto-generated** and returned in the response.

Create a simple product:
```json
{
  "familyCode": "knife"
}
```
Response: `{ "data": { "identifier": "PRD-00001", "productNumber": 1, ... } }`

Create a root product model:
```json
{
  "familyCode": "knife",
  "variantDefinitionCode": "colorThenBlade"
}
```
Response: `{ "data": { "identifier": "PRD-00003", "productNumber": 3, ... } }`

Create a child (variant or sub-model) under a parent:
```json
{
  "parentIdentifier": "PRD-00003",
  "axisValues": { "handleColor": "red" }
}
```
Response: `{ "data": { "identifier": "PRD-00003-red", "productNumber": null, ... } }`

When creating a child:
- `familyCode` is **not accepted** in the body — it's inherited from the root model.
- `variantDefinitionCode` is **not accepted** — inherited from root.
- `type` is **automatically determined**: if the child's level equals the definition's axis count, it's a `variant`; otherwise it's a `model` (sub-model).
- `level` is **automatically computed**: parent's level + 1.
- `identifier` is **automatically derived** from the parent identifier + axis value codes.
- `status` defaults to `draft`.
- `axisValues` must provide a value for every axis attribute at this child's level in the variant definition.

**`PATCH /api/v1/products/:identifier`** — Update a product. Mutable fields:
- `status` — change product status

Immutable fields (return 400 if in body): `identifier`, `productNumber`, `familyCode`, `variantDefinitionCode`, `parentIdentifier`, `type`, `level`, `axisValues`.

**Note on `axisValues` immutability:** Because the identifier is derived from axis values, changing axis values would require changing the identifier (and all children's identifiers, and all attribute_values references). This is too destructive to allow. To change a variant's axis, delete it and create a new one.

**`DELETE /api/v1/products/:identifier`** — Delete a product.
- If the product has children: fail with 409 unless `?cascade=true` (deletes all descendants and their attribute values).
- Deleting a product also deletes all its rows in `attribute_values` and `product_categories`.

### Product Children

**`GET /api/v1/products/:identifier/children`** — Get direct children of a product model. Equivalent to `GET /api/v1/products?parentIdentifier=:identifier`.

### Product Hierarchy (Full Tree)

**`GET /api/v1/products/:identifier/tree`** — Get the full variant hierarchy for a root model, returned as a nested structure:

```json
{
  "data": {
    "identifier": "PRD-00003",
    "productNumber": 3,
    "type": "model",
    "level": 0,
    "axisValues": null,
    "status": "draft",
    "children": [
      {
        "identifier": "PRD-00003-red",
        "productNumber": null,
        "type": "model",
        "level": 1,
        "axisValues": { "handleColor": "red" },
        "status": "draft",
        "children": [
          {
            "identifier": "PRD-00003-red-plain",
            "productNumber": null,
            "type": "variant",
            "level": 2,
            "axisValues": { "bladeType": "plain" },
            "status": "active",
            "children": []
          }
        ]
      }
    ]
  }
}
```

If called on a non-root product, walk up to the root and return the full tree with the requested product highlighted.

## 5. Business Rules

1. **Product identifiers are system-generated, globally unique, and immutable.** Root identifiers follow the `PRD-XXXXX` format from the product_number sequence. Child identifiers are derived from parent + axis values. Neither can be changed after creation.

2. **Every product must belong to exactly one family.** For children, the family is inherited from the root model and cannot be specified or changed.

3. **Root product models must specify a variant definition** that belongs to their family. Simple products must NOT have a variant definition. Children inherit the definition from their root.

4. **Variant definitions determine the hierarchy depth.** A definition with 2 axis levels means root (level 0) → sub-models (level 1) → variants (level 2). Children cannot be created beyond the definition's axis count.

5. **Simple products cannot have children.** Only `model` type products can have children.

6. **Children must provide axis values** for all axis attributes at their level. The values must be valid option codes for each axis attribute (query the `attribute_options` table for `singleSelect` axes, or `true`/`false` for `boolean` axes).

7. **Axis value combinations must be unique among siblings.** Enforced by the unique constraint on `(parent_identifier, axis_values)`. You cannot have two children under the same parent with identical axis values.

8. **Type is auto-determined for children:**
   - If `child_level == definition_axis_count` → type is `variant`
   - If `child_level < definition_axis_count` → type is `model` (sub-model that can have further children)

9. **Deleting a product cascades to its attribute values** in the `attribute_values` table (where `entity_type = 'product'` and `entity_id = identifier`). This is an application-level cascade, not a DB FK cascade.

10. **Status transitions:** Any status can transition to any other status. No workflow enforcement in this task — status is a simple field update. (The UI will provide guidance but the API does not restrict transitions.)

11. **A product model's variant definition cannot be changed** after children exist. This would invalidate the hierarchy structure. Return 409 if attempted.

12. **Axis values are immutable** because the identifier is derived from them. Changing axis values would cascade to identifier changes across the hierarchy and all references. To "change" a variant's axis, delete the variant and create a new one.

13. **The `product_number` sequence never resets or recycles.** If product PRD-00042 is deleted, the number 42 is not reused. The next product gets 43. This prevents confusion and ensures identifiers are never reassigned.

## 6. Resolving the Variant Definition for Any Product

Children don't store their own `variant_definition_code` — they inherit it from the root model. To resolve the definition for any product:

```typescript
async function resolveVariantDefinition(
  product: Product
): Promise<{ familyCode: string; definitionCode: string } | null> {
  if (product.type === "simple") return null;
  if (product.variantDefinitionCode) {
    return { familyCode: product.familyCode, definitionCode: product.variantDefinitionCode };
  }
  // Walk up to root
  let current = product;
  while (current.parentIdentifier) {
    current = await getProduct(current.parentIdentifier);
  }
  return current.variantDefinitionCode
    ? { familyCode: current.familyCode, definitionCode: current.variantDefinitionCode }
    : null;
}
```

This function should be implemented as a utility and cached/memoized since it's called frequently during attribute value operations.

## Implementation Addendum — Identifier Format (Phase 31 correction, 2026-03-10)

The identifier format specified in §3 above (`PRD-00001`, `PRD-00003-red-plain`) **was not implemented as written**. The actual EARL identifier format in the running application is:

| Product level | Format | Example |
|---------------|--------|---------|
| Root (simple or model) | Numeric string from `product_number` sequence | `"42"` |
| Mid-level child (sub-model) | `{parentId}_{sequence}` | `"42_1"`, `"42_2"` |
| Leaf variant | `{parentId}_{sequence}` continuing down the tree | `"42_1_1"` |
| Akeneo-imported product | Akeneo's own identifier (alphanumeric) | `"830417"`, `"LTG-001"` |

The `PRD-XXXXX` prefix and hyphen-derived child identifier scheme from the spec was not used. All documentation written after Phase 26 (Phases 27–31) that references product identifiers uses the actual format. Any examples in this document or earlier phase docs that show `PRD-00001` or `PRD-00003-red-plain` are **spec artifacts and not valid identifier values** in the running application.

## Implementation Addendum — Product Type Rename (Phase 09.6 / Phase 31 correction, 2026-03-10)

The `type` discriminator value `"simple"` specified in §2 above **was renamed to `"product"`** during implementation (Phase 09.6 — Terminology, Locales & Identifiers). All references in this document and earlier phase docs that use `type: 'simple'` or `"simple"` as a product type value are outdated.

The actual `type` enum values in the running application are:

| Value | Meaning |
|-------|---------|
| `"product"` | Standalone product with no children (was `"simple"` in the spec) |
| `"model"` | Product model that has variant children |
| `"variant"` | Leaf variant (child of a model) |

This affects: the `products.type` column, all REST API `?type=` filter parameters, the GraphQL `ProductType` enum, and any frontend type checks. Phases 27–31 use `"product"` throughout.
