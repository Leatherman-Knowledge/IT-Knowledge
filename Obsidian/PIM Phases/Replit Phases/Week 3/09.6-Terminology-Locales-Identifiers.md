# Task 9.6: Simple Type Removal, UI Terminology, Locale Format & Identifier Scheme

**Execution Plan Item:** #9.6
**Dependencies:** Products (Task 8), Product Attribute Values (Task 9), Variant Ordering (Task 9.5), Families (Task 4), Variant Definitions (Task 4.5), Locales & Channels (Task 3)
**Depended on by:** Product UI (Task 12), Akeneo Migration (Task 30)

---

## Objective

Four cross-cutting changes that affect the product layer, locale system, and import strategy. These are bundled together because they all modify foundational assumptions established in earlier phases and need to land before the product UI is built.

1. **Remove the `simple` product type** — All products live in a variant hierarchy. What was a "simple product" becomes a Top Model with a 0-axis variant definition.
2. **UI terminology** — Standardize display names for product hierarchy levels across the UI: Top Model, Model, Variant.
3. **Locale format change** — Switch from underscore (`en_US`) to hyphen (`en-US`) format for locale codes, aligning with BCP 47.
4. **Identifier scheme change** — Replace the `PRD-XXXXX` format with a numeric scheme designed for Akeneo import compatibility.

---

## 1. Remove Simple Product Type

### Problem

The `simple` product type was defined in Phase 8 as a standalone product with no variants. In practice, Leatherman's catalog does not have standalone products — every product is part of a variant hierarchy. The `simple` type adds a third code path through product creation, attribute value resolution, and category assignment without serving a real use case.

### What Changes

The `simple` value is removed from the `type` column on the `products` table. Only `model` and `variant` remain.

**What was a simple product is now a Top Model** with a variant definition that has 0 axis levels. Phase 4.5 already supports this — a variant definition with an empty axes array (`{ "axes": [] }`) is valid and places all attributes at level 0. The Top Model has no children, but it still belongs to a family and references a variant definition.

### Superseded Rules from Phase 8

> **Phase 8, Section 1 — "Simple Product" structure:** Removed. There is no standalone product type.

> **Phase 8, Section 4 — Create endpoint dispatch:** The "neither `parentIdentifier` nor `variantDefinitionCode`" path (which created a simple product) is removed. Every root product creation now requires `familyCode` and `variantDefinitionCode`.

> **Phase 8, Business Rule 3:** ~~"Simple products must NOT have a variant definition."~~ Replaced by: "Every root product (Top Model) must specify a variant definition that belongs to its family."

> **Phase 8, Business Rule 5:** ~~"Simple products cannot have children."~~ Removed. Only `model` type products can have children (unchanged), and all root products are `model` type.

> **Phase 8, Section 6 — `resolveVariantDefinition()`:** The `if (product.type === "simple") return null;` early return is removed. Every product has a resolvable variant definition.

> **Phase 8, Section 8 — Validation Schemas:** `createSimpleProductSchema` is removed. The consolidated `createProductSchema` dispatches on two paths only:
> - Has `parentIdentifier` → creating a child (Model or Variant)
> - Has `variantDefinitionCode` (no `parentIdentifier`) → creating a Top Model

### Superseded References in Other Phases

> **Phase 9, Section 1 — Attribute Level Ownership:** ~~"For simple products, all attributes are at level 0 (there is no hierarchy), so all values are written directly on the product."~~ Replaced by: "For Top Models with a 0-axis variant definition, all attributes are at level 0 and all values are written directly on the Top Model."

> **Phase 9.5, Section 1 — Sort order defaults:** ~~"On simple product and root model creation: `sort_order` defaults to `0`."~~ Replaced by: "On Top Model creation: `sort_order` defaults to `0`."

> **Phase 9.5, Section 1 — Reorder validation:** ~~"The target product must be a `model` type (not `simple` or `variant`)."~~ Replaced by: "The target product must be a `model` type (not `variant`)."

> **Phase 10, Section 1 — Category assignments:** ~~"For a simple product, categories are assigned directly on the product. For a variant hierarchy, categories are assigned on the root product model."~~ Replaced by: "Categories are assigned on the Top Model (root `model` type product). All descendants inherit the root's category assignments."

### Database Change

The `type` column CHECK constraint (if any) is updated to allow only `model` and `variant`. If the column has no CHECK constraint and validation is application-layer only, update the validation schema:

```typescript
type: z.enum(["model", "variant"])
```

## 2. UI Terminology — Product Hierarchy Display Names

### Problem

The database `type` column stores `model` for both root products and middle-level products. The UI needs distinct display names so users can tell them apart at a glance. Additionally, the terminology should align with how Leatherman thinks about its product hierarchy internally and map cleanly to EDNA and Akeneo concepts.

### Display Name Mapping

The database `type` column stays as-is. Display names are derived from `type` + whether the product has a parent:

| PIM Display Name | DB `type` | `parent_identifier` | Description |
|------------------|-----------|---------------------|-------------|
| Top Model | `model` | `NULL` | Root of a variant hierarchy. Holds level-0 attributes. |
| Model | `model` | set (has parent) | Middle axis level. Holds attributes at its level. |
| Variant | `variant` | set (has parent) | Leaf level. Holds leaf-level attributes. User-entered identifier (SKU). |

### Cross-System Terminology

For reference when building import/export and integration documentation:

| PIM | EDNA | Akeneo |
|-----|------|--------|
| Top Model | Master Product / Product | Common Product / Product Model |
| Model | N/A | Variant Axis |
| Variant | SKU | Variant |

### Implementation

Add a shared utility function that all UI surfaces use instead of displaying the raw `type` value:

```typescript
function getProductDisplayType(product: { type: string; parentIdentifier: string | null }): string {
  if (product.type === "variant") return "Variant";
  if (product.type === "model" && product.parentIdentifier === null) return "Top Model";
  if (product.type === "model" && product.parentIdentifier !== null) return "Model";
  return product.type;
}
```

This is a **UI-layer concern only**. The API continues to return the raw `type` field (`model` or `variant`) in all responses. No schema or API response changes.

## 3. Locale Format — Hyphens Instead of Underscores

### Problem

Phase 3 defined locale codes using the CLDR underscore format (`en_US`, `fr_FR`). The modern standard is BCP 47 / IETF, which uses hyphens (`en-US`, `fr-FR`). Switching now — before production data exists — avoids a painful migration later and aligns with web platform conventions (HTML `lang` attributes, HTTP `Accept-Language` headers, JavaScript `Intl` API).

### What Changes

Every surface that stores, validates, or displays locale codes switches from underscore to hyphen format.

### Validation Schema

The `localeCodeSchema` in `shared/validation.ts` updates:

```typescript
export const localeCodeSchema = z
  .string()
  .regex(/^[a-z]{2}-[A-Z]{2}$/, "Locale code must be in BCP 47 format (e.g., en-US, fr-FR)");
```

### Superseded References in Phase 3

> **Phase 3, Section 1 — `locales` table:** The `code` column description changes from `IETF format: en_US, fr_FR, de_DE` to `BCP 47 format: en-US, fr-FR, de-DE`.

> **Phase 3, Section 1 — Fallback chain example:** `fr_CA → fr_FR → en_US` becomes `fr-CA → fr-FR → en-US`.

> **Phase 3, Section 4 — Create locale example:** `"code": "en_US"` becomes `"code": "en-US"`.

> **Phase 3, Business Rule 1:** ~~"Locale codes use IETF format (`en_US`, `fr_FR`)"~~ Replaced by: "Locale codes use BCP 47 format (`en-US`, `fr-FR`) — not camelCase."

### Affected Tables and Columns

| Table | Column(s) | Change |
|-------|-----------|--------|
| `locales` | `code` (PK), `fallback_locale_code` (FK) | Values use hyphen format |
| `channels` | `labels` (JSONB keys), `locale_codes` (JSONB array values) | Keys and values use hyphen format |
| `families` | `labels` (JSONB keys) | Keys use hyphen format |
| `family_variant_definitions` | `labels` (JSONB keys) | Keys use hyphen format |
| `attributes` | `labels` (JSONB keys) | Keys use hyphen format |
| `attribute_groups` | `labels` (JSONB keys) | Keys use hyphen format |
| `attribute_options` | `labels` (JSONB keys) | Keys use hyphen format |
| `categories` | `labels` (JSONB keys) | Keys use hyphen format |
| `category_trees` | `labels` (JSONB keys) | Keys use hyphen format |
| `attribute_values` | `locale_code` (FK) | Values use hyphen format |
| `family_attribute_requirements` | `locale_code` | Values use hyphen format |

### Superseded Reference in Phase 9.5

> **Phase 9.5, Business Rule 4:** ~~"Locale codes are exempt from camelCase. They follow the IETF/CLDR `ll_CC` convention (e.g., `en_US`)."~~ Replaced by: "Locale codes are exempt from camelCase. They follow the BCP 47 `ll-CC` convention (e.g., `en-US`), which includes hyphens and uppercase characters."

### Migration

Since this is pre-production with no live data, the migration is straightforward:

1. Update all seed data to use hyphen format.
2. Update any existing locale records: `UPDATE locales SET code = REPLACE(code, '_', '-'), fallback_locale_code = REPLACE(fallback_locale_code, '_', '-');`
3. Update all JSONB label keys across all tables that store locale-keyed labels.
4. Update `channels.locale_codes` arrays.
5. Update `attribute_values.locale_code` and `family_attribute_requirements.locale_code`.

If seed data is the only data, dropping and recreating with the new format is simpler than running UPDATE statements.

## 4. Identifier Scheme — Akeneo-Compatible Product Codes

### Problem

Phase 8 defined product identifiers as `PRD-XXXXX` (zero-padded sequential numbers with a prefix) for root products, and `{parent}-{axisValue1}-{axisValue2}` for children. This format doesn't align with Akeneo's existing product codes, which the migration (Task 30) needs to preserve for Top Models and Variants. A new scheme is needed that:

- Allows imported Top Model codes to be preserved as-is (they are plain integers in Akeneo)
- Allows imported Variant codes (SKUs) to be preserved as-is
- Generates clean, predictable codes for Models (middle level) that follow a parent-child pattern
- Lets the auto-increment sequence resume above the max imported value

### Identifier Formats by Product Level

#### Top Models (root `model` type, no parent)

| Aspect | Value |
|--------|-------|
| **Format** | Plain integer: `10`, `11`, `12`, ... |
| **Source** | `product_number` identity column, converted to string |
| **Column type** | `identifier` remains `VARCHAR(255)` PK |
| **Auto-generation** | System-generated. User does not supply an identifier. |
| **Akeneo import** | Imported Top Models keep their exact Akeneo code. The `product_number` is set to match. |

The generation process:
1. Insert the product row. The `product_number` column auto-increments via `GENERATED BY DEFAULT AS IDENTITY`.
2. Set the `identifier` column to `productNumber.toString()` (no prefix, no padding).

```typescript
function generateTopModelIdentifier(productNumber: number): string {
  return productNumber.toString();
}
```

> **Supersedes Phase 8, Section 3 — Root identifier format:** ~~`PRD-{number padded to 5 digits}`~~ Replaced by plain integer string.

> **Supersedes Phase 8, Section 3 — `generateRootIdentifier()`:** The function drops the `PRD-` prefix and zero-padding.

#### Models (middle axis level, `model` type with parent)

| Aspect | Value |
|--------|-------|
| **Format** | `{parent_identifier}_{incrementing_integer}`: `10_1`, `10_2`, `10_3` |
| **Separator** | Underscore (`_`) |
| **Auto-generation** | System-generated from parent identifier + next sibling number. |
| **Akeneo import** | Middle Models are regenerated to this format. They do not preserve Akeneo codes. |

The generation process:
1. Query existing siblings of the same parent.
2. Extract the numeric suffix from each sibling's identifier (the part after the last `_`).
3. Set the new identifier to `{parentIdentifier}_{max_suffix + 1}`. If no siblings exist, start at `1`.

```typescript
function generateModelIdentifier(
  parentIdentifier: string,
  existingSiblingIdentifiers: string[]
): string {
  if (existingSiblingIdentifiers.length === 0) {
    return `${parentIdentifier}_1`;
  }
  const maxSuffix = existingSiblingIdentifiers
    .map(id => parseInt(id.split('_').pop()!, 10))
    .reduce((max, n) => Math.max(max, n), 0);
  return `${parentIdentifier}_${maxSuffix + 1}`;
}
```

For two-level hierarchies, the pattern nests: a level-2 Model under `10_1` would be `10_1_1`, `10_1_2`, etc.

> **Supersedes Phase 8, Section 3 — Child identifier format:** ~~`{parent_identifier}-{axisValue1}-{axisValue2}`~~ For Models, replaced by `{parent_identifier}_{incrementing_integer}`.

#### Variants (leaf level, `variant` type)

| Aspect | Value |
|--------|-------|
| **Format** | User-entered string (typically a SKU) |
| **Auto-generation** | **None.** The user must provide the identifier. |
| **Akeneo import** | Variants keep their exact Akeneo codes (SKUs). |

This is a significant change from Phase 8, where identifiers were always system-generated and the API did not accept an `identifier` field in the create body.

**Create endpoint change:** When creating a Variant (a child whose level equals the variant definition's axis count), the request body must include an `identifier` field:

```json
{
  "parentIdentifier": "10_1",
  "axisValues": { "bladeType": "plain" },
  "identifier": "832-RED-PLAIN"
}
```

**Validation:**
- `identifier` is **required** when creating a Variant. Return 400 if missing.
- `identifier` is **rejected** when creating a Top Model or Model. Return 400 if provided.
- Must be globally unique (enforced by the PK constraint). Return 409 if a product with that identifier already exists.
- Must be a non-empty string. Maximum 255 characters (column width).

> **Supersedes Phase 8, Section 3 — "Users never supply an identifier":** This rule now applies only to Top Models and Models. Variant identifiers are user-supplied.

> **Supersedes Phase 8, Section 3 — `generateChildIdentifier()`:** For Variants, the function is not called. The user-supplied identifier is used directly.

### Summary of Identifier Changes vs Phase 8

| Product Level | Phase 8 Format | Phase 9.6 Format | Auto-generated? |
|---------------|----------------|-------------------|-----------------|
| Top Model | `PRD-00001` | `10` | Yes (sequence) |
| Model (middle) | `PRD-00003-red` (parent + axis values) | `10_1` (parent + increment) | Yes (sibling increment) |
| Variant (leaf) | `PRD-00003-red-plain` (parent + axis values) | User-entered (e.g., `832-RED-PLAIN`) | No — user provides |

### Example Hierarchy with New Identifiers

```
[Top Model: 10]                                    ← type: model, level: 0
  family: knife
  variant definition: colorThenBlade
  ├── [Model: 10_1]                                ← type: model, level: 1, axis: { handleColor: "red" }
  │   ├── [Variant: 832-RED-PLAIN]                 ← type: variant, level: 2, axis: { bladeType: "plain" }
  │   └── [Variant: 832-RED-SERRATED]              ← type: variant, level: 2, axis: { bladeType: "serrated" }
  ├── [Model: 10_2]                                ← type: model, level: 1, axis: { handleColor: "blue" }
  │   ├── [Variant: 832-BLUE-PLAIN]                ← type: variant, level: 2, axis: { bladeType: "plain" }
  │   └── [Variant: 832-BLUE-SERRATED]             ← type: variant, level: 2, axis: { bladeType: "serrated" }
```

### One-Level Variant Definition Example

For a Top Model with a single axis level (no middle Models), Variants are direct children:

```
[Top Model: 11]                                    ← type: model, level: 0
  family: multitool
  variant definition: colorOnly
  ├── [Variant: WINGMAN-BLK]                       ← type: variant, level: 1, axis: { handleColor: "black" }
  ├── [Variant: WINGMAN-SLV]                       ← type: variant, level: 1, axis: { handleColor: "silver" }
  └── [Variant: WINGMAN-RED]                       ← type: variant, level: 1, axis: { handleColor: "red" }
```

### Schema Changes

The `product_number` column remains `GENERATED BY DEFAULT AS IDENTITY`. The `BY DEFAULT` clause (vs `ALWAYS`) allows explicit values to be inserted during the Akeneo import, overriding the sequence.

The `identifier` column remains `VARCHAR(255)` PK. No type change needed — it stores string representations of integers for Top Models, underscore-delimited strings for Models, and arbitrary strings for Variants.

### Superseded Validation Schemas

> **Phase 8, Section 8:** The `createProductSchema` changes:
> - Top Model path: `familyCode` (required), `variantDefinitionCode` (required), optional `status`. No `identifier` field.
> - Child path: `parentIdentifier` (required), `axisValues` (required), optional `status`. Plus conditional `identifier`: required if the child will be a Variant (leaf level), rejected if the child will be a Model.

The route handler must resolve whether the child is a Variant or Model before validating the `identifier` field. This requires looking up the parent's variant definition to determine the axis count, then comparing `parent.level + 1` to the axis count.

### Superseded Business Rules from Phase 8

> **Phase 8, Business Rule 1:** ~~"Root identifiers follow the `PRD-XXXXX` format."~~ Replaced by: "Top Model identifiers are plain integers from the product_number sequence."

> **Phase 8, Business Rule 1:** ~~"Child identifiers are derived from parent + axis values."~~ Replaced by: "Model identifiers follow the `{parent}_{increment}` pattern. Variant identifiers are user-supplied."

> **Phase 8, Business Rule 12:** ~~"Axis values are immutable because the identifier is derived from them."~~ Axis values remain immutable, but the reason changes: for Models, the identifier is no longer derived from axis values (it uses an incrementing number). Axis values are still immutable because they define the semantic meaning of the variant branch and are stored in the `axis_values` JSONB column. The unique constraint on `(parent_identifier, axis_values)` still prevents duplicate branches.

### Akeneo Import Sequence Seeding

After importing all Akeneo products, seed the `product_number` sequence to resume above the maximum imported value:

```sql
SELECT setval('products_product_number_seq', (SELECT MAX(product_number) FROM products));
```

This ensures the next auto-generated Top Model gets a number above all imported ones. The import process should:

1. Import Top Models with their exact Akeneo numeric codes, explicitly setting `product_number` and `identifier`.
2. Import Models (middle level) using the new `{parent}_{increment}` format — these are regenerated, not preserved from Akeneo.
3. Import Variants with their exact Akeneo codes (SKUs) as the `identifier`.
4. After all imports complete, run the `setval` statement above.

## 7. Business Rules

1. **No simple products.** Every product is either a Top Model (root `model`), a Model (child `model`), or a Variant (child `variant`). A product that would have been "simple" is a Top Model with a 0-axis variant definition.

2. **Every Top Model must specify a variant definition.** The `variantDefinitionCode` field is required on root product creation. A 0-axis definition is valid for products with no variants.

3. **UI display names are derived, not stored.** The database `type` column stores `model` or `variant`. The display name (Top Model, Model, Variant) is computed from `type` + `parentIdentifier` at the UI layer.

4. **Locale codes use BCP 47 hyphen format.** `en-US`, `fr-FR`, `de-DE` — not underscores. Locale codes remain exempt from camelCase validation.

5. **Top Model identifiers are system-generated integers.** Users do not supply identifiers for Top Models. The `product_number` sequence provides the value.

6. **Model identifiers are system-generated with parent prefix.** Format: `{parentIdentifier}_{incrementingInteger}`. Users do not supply identifiers for Models.

7. **Variant identifiers are user-supplied.** The create endpoint requires an `identifier` field when creating a Variant. This is typically a SKU or product code from the source system.

8. **The `product_number` sequence never resets or recycles.** Deleting Top Model `42` does not free up the number. The next Top Model gets `43`.

9. **Axis values remain immutable** even though Model identifiers are no longer derived from them. Axis values define the semantic meaning of a variant branch and are enforced unique per parent via the `(parent_identifier, axis_values)` constraint.

10. **Akeneo import preserves Top Model and Variant identifiers.** Middle Models are regenerated to the `{parent}_{increment}` format. The sequence is seeded after import.
