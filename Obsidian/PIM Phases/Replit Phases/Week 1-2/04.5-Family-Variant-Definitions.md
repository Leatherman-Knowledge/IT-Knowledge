# Task 4.5: Family Variant Definitions

**Execution Plan Item:** #4.5
**Dependencies:** Families (Task 4)
**Depended on by:** Products & Product Models (variant hierarchy creation)

---

## Objective

Add a **variant definition** layer to families so that a single family can support multiple variant configurations. A variant definition specifies how many axes a product has, which attributes serve as axes at each level, and which of the family's attributes live at which level in that configuration.

Phase 4 placed `variant_level` directly on `family_attributes`, forcing every product in a family into the same variant structure. This phase replaces that with a flexible system where each family can have multiple variant definitions, and each product references the one it uses.

---

## 1. Core Concepts

### What is a Variant Definition?

A **variant definition** is a named variant configuration within a family. It answers three questions:

1. **How many axis levels does the product hierarchy have?** (0 = simple product, 1 = one axis, 2 = two axes, etc.)
2. **Which attribute(s) serve as the axis at each level?** These are the "dimension-defining" attributes whose values create variant branches.
3. **Which of the family's attributes live at which level?** Each attribute is either at level 0 (common/parent) or at a specific variant level.

### Multiple Axes Per Level

A single axis level can have **multiple axis attributes**. When this happens, variant branches at that level are the **cartesian product** of all axis attribute values. For example:

- Level 1 with axes `[handleColor, sheathType]` where handleColor has values {red, blue} and sheathType has values {leather, nylon} produces 4 variant groups: (red, leather), (red, nylon), (blue, leather), (blue, nylon).
- Level 2 with axis `[bladeType]` where bladeType has values {plain, serrated} then produces 2 individual variants under each level-1 group.

### Axis Attributes

An axis attribute is special:
- It must be an **enumerable type** (`singleSelect` or `boolean`) — types whose values can serve as variant dimension identifiers.
- Its values determine what variant branches exist at its level.
- An axis attribute is implicitly assigned to its axis level (its values are set at that level in the hierarchy).
- An attribute can only be an axis at **one** level per definition.

### How Attribute Levels Are Resolved

For a given variant definition, every family attribute is at exactly one level:

1. **Axis attributes** → their level comes from `family_variant_axes.level`
2. **Non-axis attributes explicitly assigned** → their level comes from `family_variant_attribute_levels.variant_level`
3. **All other attributes** → default to **level 0** (common/parent)

## 2. Database Schema

### Modified: `family_attributes`

**Remove** the `variant_level` column and the `idx_fa_variant_level` index. Variant levels are now per-definition, not per-family.

The table retains: `family_code`, `attribute_code`, `sort_order`, `created_at`, composite PK `(family_code, attribute_code)`, and index `idx_fa_family`.

### Table: `family_variant_definitions`

Header table for each named variant configuration within a family.

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `family_code` | `VARCHAR(100)` | FK → families.code, ON DELETE CASCADE | |
| `code` | `VARCHAR(100)` | NOT NULL | camelCase identifier, unique within family |
| `labels` | `JSONB` | NOT NULL, DEFAULT '{}' | `{ "en_US": "Color then Blade", "fr_FR": "Couleur puis lame" }` |
| `sort_order` | `INTEGER` | NOT NULL, DEFAULT 0 | Display order within the family |
| `created_at` | `TIMESTAMPTZ` | NOT NULL, DEFAULT now() | |
| `updated_at` | `TIMESTAMPTZ` | NOT NULL, DEFAULT now() | |
| PRIMARY KEY | | `(family_code, code)` | |

**Index:** `CREATE INDEX idx_fvd_family ON family_variant_definitions(family_code);`

### Table: `family_variant_axes`

Defines which attribute(s) are axes at each level in a definition. A level can have **multiple axis attributes** — variant branches at that level are the cartesian product of all axis attribute values.

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `family_code` | `VARCHAR(100)` | NOT NULL | |
| `definition_code` | `VARCHAR(100)` | NOT NULL | |
| `level` | `INTEGER` | NOT NULL | 1-indexed (1 = first axis level, 2 = second, etc.) |
| `attribute_code` | `VARCHAR(255)` | FK → attributes.code, ON DELETE CASCADE, NOT NULL | The axis attribute |
| `sort_order` | `INTEGER` | NOT NULL, DEFAULT 0 | Display order within a level (when multiple axes at same level) |
| PRIMARY KEY | | `(family_code, definition_code, level, attribute_code)` | Multiple attributes per level |
| UNIQUE | | `(family_code, definition_code, attribute_code)` | An attribute can only be an axis once per definition |

**Foreign keys:**
- `(family_code, definition_code)` → `family_variant_definitions(family_code, code)` ON DELETE CASCADE
- `(family_code, attribute_code)` → `family_attributes(family_code, attribute_code)` ON DELETE CASCADE

**Indexes:**
- `CREATE INDEX idx_fva_definition ON family_variant_axes(family_code, definition_code);`

### Table: `family_variant_attribute_levels`

Assigns non-axis attributes to non-zero levels per definition. Attributes **not** in this table (and not axes) default to level 0 (common).

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `family_code` | `VARCHAR(100)` | NOT NULL | |
| `definition_code` | `VARCHAR(100)` | NOT NULL | |
| `attribute_code` | `VARCHAR(255)` | NOT NULL | |
| `variant_level` | `INTEGER` | NOT NULL | 1+ only (level 0 = default, no row needed) |
| PRIMARY KEY | | `(family_code, definition_code, attribute_code)` | |

**Foreign keys:**
- `(family_code, definition_code)` → `family_variant_definitions(family_code, code)` ON DELETE CASCADE
- `(family_code, attribute_code)` → `family_attributes(family_code, attribute_code)` ON DELETE CASCADE

**Indexes:**
- `CREATE INDEX idx_fval_definition_level ON family_variant_attribute_levels(family_code, definition_code, variant_level);`

## 3. API Endpoints

### Variant Definitions CRUD

**`GET /api/v1/families/:code/variant-definitions`** — List all variant definitions for a family.
Returns code, labels, axis level count, and a summary of axes per level.

**`GET /api/v1/families/:code/variant-definitions/:defCode`** — Get a single variant definition with its full structure: axes per level and all family attributes grouped by their resolved level.

Response structure:
```json
{
  "familyCode": "knife",
  "code": "colorThenBlade",
  "labels": { "en_US": "Color then Blade" },
  "sortOrder": 1,
  "axes": [
    { "level": 1, "attributes": [{ "code": "handleColor", "type": "singleSelect" }] },
    { "level": 2, "attributes": [{ "code": "bladeType", "type": "singleSelect" }] }
  ],
  "attributesByLevel": {
    "0": [
      { "code": "productName", "type": "text", "isAxis": false },
      { "code": "longDescription", "type": "textarea", "isAxis": false },
      { "code": "mainImage", "type": "mediaFile", "isAxis": false },
      { "code": "weight", "type": "measurement", "isAxis": false }
    ],
    "1": [
      { "code": "handleColor", "type": "singleSelect", "isAxis": true },
      { "code": "colorImage", "type": "mediaFile", "isAxis": false }
    ],
    "2": [
      { "code": "bladeType", "type": "singleSelect", "isAxis": true },
      { "code": "sku", "type": "text", "isAxis": false },
      { "code": "price", "type": "price", "isAxis": false }
    ]
  }
}
```

**`POST /api/v1/families/:code/variant-definitions`** — Create a variant definition.
```json
{
  "code": "colorThenBlade",
  "labels": { "en_US": "Color then Blade" }
}
```
Starts with 0 axes and all attributes at level 0.

**`PATCH /api/v1/families/:code/variant-definitions/:defCode`** — Update metadata (labels, sort_order). Code is immutable (400 if `code` in body).

**`DELETE /api/v1/families/:code/variant-definitions/:defCode`** — Delete a variant definition. Cascades to axes and attribute-level assignments. (Future: will need to check for products referencing this definition before allowing deletion.)

### Variant Axes

**`PUT /api/v1/families/:code/variant-definitions/:defCode/axes`** — Bulk-replace all axes for a definition.

```json
{
  "axes": [
    { "level": 1, "attributeCodes": ["handleColor", "sheathType"] },
    { "level": 2, "attributeCodes": ["bladeType"] }
  ]
}
```

This is a full replace: deletes all existing axes, then inserts the new set.

**Validation:**
- Each attribute must be assigned to the family.
- Each attribute must be an axis-eligible type (`singleSelect` or `boolean`).
- Levels must be sequential starting at 1 (no gaps).
- An attribute cannot appear at more than one level.
- Maximum 5 axis levels per definition.

**Side effect:** When axes change, any entries in `family_variant_attribute_levels` referencing levels that no longer exist are deleted. For example, if the definition previously had 2 axis levels and you change it to 1, any attribute-level assignments at level 2 are removed.

### Variant Attribute Levels

**`GET /api/v1/families/:code/variant-definitions/:defCode/attribute-levels`** — Get all attribute level assignments for this definition.

Returns the complete picture: every family attribute with its resolved level (computed from axes table + attribute-levels table + defaults). Same data as the `attributesByLevel` in the GET definition detail, but as a flat list.

**`PUT /api/v1/families/:code/variant-definitions/:defCode/attribute-levels`** — Bulk-replace non-axis attribute level assignments.

```json
{
  "attributeLevels": [
    { "attributeCode": "colorImage", "variantLevel": 1 },
    { "attributeCode": "sku", "variantLevel": 2 },
    { "attributeCode": "price", "variantLevel": 2 }
  ]
}
```

Attributes not included default to level 0. Only non-axis attributes can be included.

**Validation:**
- Each attribute must be assigned to the family.
- Each attribute must NOT be an axis attribute in this definition (axes get their level from the axes table).
- `variantLevel` must be between 1 and the definition's axis level count. (Level 0 attributes are simply omitted from the body.)

## 4. Business Rules

1. **Definition codes are camelCase, scoped to family, and immutable.** PK is `(family_code, code)`. Validated by the standard `camelCaseCodeSchema` regex on create. PATCH returns 400 if `code` is in the body.

2. **Axis attributes must be assigned to the family.** FK from `family_variant_axes(family_code, attribute_code)` to `family_attributes` enforces this.

3. **Axis attributes must be enumerable types.** Only `singleSelect` and `boolean` are valid axis types. These are the types whose distinct values can serve as variant dimension identifiers. Validate on PUT axes.

4. **Axis levels must be sequential starting at 1.** No gaps allowed. If there are 2 axis levels, they must be level 1 and level 2.

5. **Max 5 axis levels per definition.** Practical upper bound. Real-world products rarely exceed 3.

6. **A level can have multiple axis attributes.** Variant branches at that level are the cartesian product of all axis values at that level. E.g., axes `[handleColor, sheathType]` at level 1 produces `color × sheath` combinations.

7. **An attribute can only be an axis once per definition.** Enforced by the UNIQUE constraint `(family_code, definition_code, attribute_code)`. An attribute cannot appear as an axis at two different levels.

8. **Non-axis attribute levels must be within range.** Level must be between 1 and the definition's axis level count. You can't put an attribute at level 3 if the definition only has 2 axis levels. Level 0 attributes are simply omitted from the PUT body.

9. **Axis attributes cannot appear in the attribute-levels table.** Their level is defined by the axes table. Validate on PUT to attribute-levels — return 400 if an axis attribute is included.

10. **Removing an attribute from a family cascades to all definitions.** FKs on `family_variant_axes` and `family_variant_attribute_levels` reference `family_attributes`, so ON DELETE CASCADE removes the attribute from all definitions automatically.

11. **Deleting a definition cascades.** ON DELETE CASCADE removes its axes and attribute-level assignments.

12. **A definition with 0 axis levels is valid.** Represents simple (non-variant) products where all attributes are at level 0. The axes body would be `{ "axes": [] }`.

## 6. Example: The "Knife" Family

```
Family: knife
Labels: { "en_US": "Knife" }
Label attribute: productName
Image attribute: mainImage

Attributes (pool — no variant levels here, just the attribute set):
  - productName (text, localizable)
  - longDescription (textarea, localizable, channelizable)
  - mainImage (mediaFile)
  - weight (measurement)
  - bladeSteel (entityReference)
  - productLine (singleSelect)
  - handleColor (singleSelect)
  - sheathType (singleSelect)
  - colorImage (mediaFile)
  - bladeType (singleSelect)
  - sku (text, unique)
  - price (price, channelizable)

─────────────────────────────────────────────────────

Variant Definition: "colorThenBlade" — 2 axis levels
  Axes:
    Level 1: [handleColor]
    Level 2: [bladeType]
  Attribute levels:
    Level 0 (common): productName, longDescription, mainImage,
                       weight, bladeSteel, productLine, sheathType
    Level 1: handleColor (axis), colorImage
    Level 2: bladeType (axis), sku, price

─────────────────────────────────────────────────────

Variant Definition: "colorSheathThenBlade" — 2 axis levels (multi-axis at level 1)
  Axes:
    Level 1: [handleColor, sheathType]
    Level 2: [bladeType]
  Attribute levels:
    Level 0 (common): productName, longDescription, mainImage,
                       weight, bladeSteel, productLine
    Level 1: handleColor (axis), sheathType (axis), colorImage
    Level 2: bladeType (axis), sku, price
  → Level 1 produces handleColor × sheathType combinations

─────────────────────────────────────────────────────

Variant Definition: "bladeOnly" — 1 axis level
  Axes:
    Level 1: [bladeType]
  Attribute levels:
    Level 0 (common): productName, longDescription, mainImage,
                       weight, bladeSteel, productLine,
                       handleColor, sheathType, colorImage
    Level 1: bladeType (axis), sku, price

─────────────────────────────────────────────────────

Variant Definition: "simple" — 0 axis levels
  Axes: (none)
  Level 0: all attributes
```
