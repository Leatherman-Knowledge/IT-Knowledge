# Task 7.5: Translations Management System

**Execution Plan Item:** #7.5 (refinement of Task 7 UI)
**Dependencies:** UI Shell (Task 7), Locales (Task 3), all entity types with labels (Tasks 2–5)
**Origin:** Replit built per-locale label inputs on every entity form. This was refactored into a centralized translations page so entity forms stay clean and translation is treated as its own workflow.

---

## Overview

The Translations system centralizes all label translation management into a single, dedicated page. Previously, every entity create/edit dialog included per-locale label inputs for all active locales. This was replaced with a streamlined approach: entity forms now only capture the required English (en_US) label, while a purpose-built Translations page provides a spreadsheet-like grid editor for managing all locale translations in bulk.

---

## Design Decisions

### Why Centralize Translations?

1. **Reduced form complexity** — Entity create/edit dialogs were cluttered with one text input per active locale. For a system with 10+ locales, this meant 10+ fields just for labels, making forms unwieldy.
2. **Separation of concerns** — Creating an attribute and translating its label into French are conceptually different tasks, likely performed by different people (developers vs. translators).
3. **Batch workflow** — Translators work across many entities at once. A grid view with inline editing is far more efficient than opening individual entity dialogs one by one.
4. **Coverage visibility** — A centralized view makes it easy to see at a glance which entities still need translations and for which locales.

### Entity Types Covered

The system aggregates labels from 8 entity types:

| Entity Type | DB Table | Parent Relationship |
|---|---|---|
| `attribute` | `attributes` | — |
| `attributeGroup` | `attribute_groups` | — |
| `attributeOption` | `attribute_options` | `parentCode` = `attributeCode` |
| `family` | `families` | — |
| `familyVariantDefinition` | `family_variant_definitions` | `parentCode` = `familyCode` |
| `channel` | `channels` | — |
| `categoryTree` | `category_trees` | — |
| `category` | `categories` | `parentCode` = `treeCode` |

### Scope Boundary

This page manages **entity label translations** only — the human-readable display names of structural entities like attributes, families, categories, etc. It does **not** manage product attribute values. Localizable product attribute values (e.g., the French product name for a specific product) are managed through the product edit page's channel/locale picker (Task 12).

## What Was Built

### 1. Simplified LocaleLabelsEditor Component

**File:** `client/src/components/locale-labels-editor.tsx`

The shared `LocaleLabelsEditor` component, used in all entity create/edit forms, was reduced from a multi-locale input group to a single English-only input:

- Shows only `en_US` (required) with a star indicator
- Includes helper text directing users to the Translations page for other locales
- All entity forms (attributes, attribute groups, families, channels, category trees, categories, variant definitions, attribute options) automatically inherit this simplification since they all use the shared component

### 2. Translations Page (Frontend)

**Route:** `/translations`
**File:** `client/src/pages/translations.tsx`
**Sidebar Location:** Settings group, with a Languages icon

A full-featured translation management page with the following capabilities:

#### Filter Bar
- **Search input** — Filters entities by code or label text (searches across all locale values)
- **Entity type dropdown** — Filter to show only one entity type (e.g., "Attributes", "Families")
- **Clear filters button** — Resets all active filters

#### Translation Grid
- **Horizontally scrollable table** — Handles any number of locales without breaking layout
- **Sticky columns** — The first three columns (Type, Code, en_US) are sticky on the left so they remain visible during horizontal scroll
- **Type column** — Shows a badge with the human-readable entity type name
- **Code column** — Displays the entity code; for child entities (options, variant definitions, categories), also shows the parent code prefix
- **en_US column** — Read-only display of the primary English label
- **Locale columns** — One column per active non-English locale, each showing the current translation or "Click to translate" placeholder

#### Inline Editing
- **Click to edit** — Clicking any locale cell enters inline edit mode with a text input
- **Keyboard shortcuts** — `Enter` to confirm, `Escape` to cancel
- **Confirm/Cancel buttons** — Small icon buttons alongside the input for mouse users
- **Pending changes tracking** — Edits are held in local state (a `Map<string, PendingChange>`) until explicitly saved; cells with pending changes are highlighted with an amber background

#### Batch Save
- **Pending count badge** — Appears in the page header showing how many unsaved changes exist
- **Save All button** — Submits all pending changes to the backend in a single batch request
- **Discard button** — Clears all pending changes without saving
- **Post-save cache invalidation** — After save, invalidates query caches for translations and all entity types so that entity list pages reflect updated labels

#### Coverage Statistics
- **Stats card** — Shows `translated/total` count and a percentage badge
- **Dynamic calculation** — Counts how many (entity × non-primary-locale) cells have non-empty values, including any pending changes
- **Scope-aware** — Stats update based on the current filter (e.g., showing coverage for just "Attributes" when that filter is active)

#### Pagination
- Uses the shared `TablePagination` component with configurable page sizes (default 25 rows)

### 3. Backend API Endpoints

**File:** `server/routes.ts`

#### `GET /api/v1/translations`

Aggregates all translatable entities from across 8 database tables into a single flat list.

**Response shape:**
```json
{
  "data": [
    {
      "entityType": "attribute",
      "code": "color",
      "labels": { "en_US": "Color", "fr_FR": "Couleur" }
    },
    {
      "entityType": "attributeOption",
      "code": "red",
      "parentCode": "color",
      "labels": { "en_US": "Red" }
    }
  ]
}
```

The endpoint calls existing storage methods where available (`getAttributes`, `getAttributeGroups`, `getFamilies`, `getChannels`, `getCategoryTrees`) and three new storage methods for tables that previously lacked a "get all" query (`getAllAttributeOptions`, `getAllVariantDefinitions`, `getAllCategories`).

#### `POST /api/v1/translations/batch`

Accepts a batch of label updates and applies them one by one.

**Request body:**
```json
{
  "updates": [
    {
      "entityType": "attribute",
      "code": "color",
      "locale": "fr_FR",
      "value": "Couleur"
    }
  ]
}
```

**Validation rules:**
- `updates` must be a non-empty array with a maximum of 500 items
- Each update's `entityType` must be one of the 8 valid types
- `code` and `locale` must be non-empty strings
- `value` must be a string (empty string removes the translation)
- `parentCode` is required for `attributeOption` and `familyVariantDefinition` entity types

**Response:**
```json
{
  "data": {
    "updated": 2,
    "requested": 2
  }
}
```

All batch saves are recorded in the audit log.

### 4. Storage Layer Additions

**File:** `server/storage.ts`

Four new methods added to `IStorage` and `DatabaseStorage`:

| Method | Purpose |
|--------|---------|
| `getAllAttributeOptions()` | Returns all attribute options across all attributes |
| `getAllVariantDefinitions()` | Returns all variant definitions across all families |
| `getAllCategories()` | Returns all categories across all trees |
| `updateEntityLabel(entityType, code, locale, value, parentCode?)` | Generic label updater — reads current labels JSONB, merges locale, writes back |

---

## Data Flow

```
User clicks locale cell → Local editingCells state updated →
User confirms edit → pendingChanges Map updated (amber highlight) →
User clicks "Save All" → POST /api/v1/translations/batch →
Backend validates → updateEntityLabel() for each item →
JSONB labels column updated in appropriate table →
Response with counts → Frontend clears pending changes →
Cache invalidated → Grid refreshes with saved values
```

## Key Technical Details

### Cell Key Format
Each cell is uniquely identified by: `{entityType}:{parentCode/}{code}:{localeCode}`

Examples:
- `attribute:color:fr_FR`
- `attributeOption:color/red:fr_FR`
- `category:main-tree/tools:de_DE`

### Empty Value Handling
- Empty string on save removes the locale key from the JSONB object entirely
- Frontend displays "Click to translate" in italicized muted text for empty cells
- Coverage statistics count a cell as "translated" only if it has a non-empty value
