# Phase 20.6: API Key Authentication & Machine-to-Machine Access

**Execution Plan Item:** #20.6  
**Replit Phase:** 20.6 (Week 4 — addendum)  
**Dependencies:** Phase 01 (Auth & RBAC schema), Phase 17 (API consistency & Swagger), Phase 19 (RBAC enforcement & `requirePermission`)  
**Depended on by:** External integrations (ERP, e-commerce, data pipelines), Akeneo import (Execution Plan 30), any future GraphQL consumers  
**Supersedes:** Phase 12.4 (Service-to-Service Auth — OAuth 2.0 Client Credentials). That phase is now **cancelled**; API keys are simpler, self-contained, and avoid additional Entra ID configuration. If full OAuth 2.0 Client Credentials become necessary later, it can be revisited as a separate phase.

---

## Objective

Enable **machine-to-machine (M2M) access** to both the REST API and GraphQL API via API key authentication. API keys are **standalone identities** — they are not linked to a user account. Each key is assigned its own roles directly using the existing RBAC role system from Phase 01, so permissions are scoped independently per key. External systems (ERP, e-commerce platforms, data pipelines, import scripts) can call the API without browser-based SSO or any dependency on a `pim_users` record.

## 1. Database Schema

### Table: `api_keys`

Add to `shared/schema.ts` alongside the existing auth tables.

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `id` | `SERIAL` | PK | Auto-increment identifier |
| `name` | `VARCHAR(255)` | NOT NULL | Human-readable label (e.g. "ERP Sync", "Shopify Integration") |
| `description` | `TEXT` | NULL | Optional notes about what this key is used for |
| `key_hash` | `VARCHAR(255)` | NOT NULL, UNIQUE | SHA-256 hash of the full API key — plaintext is never stored |
| `key_prefix` | `VARCHAR(12)` | NOT NULL | First 8 characters of the key for UI identification (e.g. `earl_a3f8…`) |
| `expires_at` | `TIMESTAMPTZ` | NULL | Optional expiry; NULL means no expiry |
| `last_used_at` | `TIMESTAMPTZ` | NULL | Updated on each successful authentication |
| `is_active` | `BOOLEAN` | NOT NULL, DEFAULT `true` | Soft-disable without deleting |
| `created_by` | `VARCHAR(255)` | NULL, FK → `pim_users.id` | The admin who created this key (for audit; NULL if created via seed/migration) |
| `created_at` | `TIMESTAMPTZ` | NOT NULL, DEFAULT `now()` | |
| `updated_at` | `TIMESTAMPTZ` | NOT NULL, DEFAULT `now()` | |

### Table: `api_key_roles`

Maps API keys to roles. Mirrors the structure of `user_roles` but for API keys.

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `api_key_id` | `INTEGER` | FK → `api_keys.id`, ON DELETE CASCADE | |
| `role_code` | `VARCHAR(100)` | FK → `roles.code`, ON DELETE CASCADE | |
| `assigned_at` | `TIMESTAMPTZ` | NOT NULL, DEFAULT `now()` | |
| PRIMARY KEY | | `(api_key_id, role_code)` | |

This reuses the existing `roles` table — the same admin, editor, viewer (and any custom) roles that apply to users also apply to API keys. An API key's effective permissions are determined by its assigned roles, resolved through `role_permissions` → `permissions` exactly like user permissions.

### Key Format

Generated keys follow the pattern `earl_<32 random hex chars>` (total ~37 characters). The `earl_` prefix makes keys identifiable in logs and config files. Example: `earl_a3f8b2c1d4e5f67890abcdef12345678`.

### Drizzle Additions

- Define both tables in `shared/schema.ts` using the existing Drizzle patterns.
- Add Zod insert/select schemas and TypeScript types.
- Push schema to DB.

## 2. Storage Methods

Add to `IStorage` interface and `DatabaseStorage` class in `server/storage.ts`, following the same patterns as existing entity storage methods.

| Method | Signature | Description |
|--------|-----------|-------------|
| `getApiKeyByHash` | `(keyHash: string) → ApiKeyWithRoles \| undefined` | Lookup by hash; returns key record with its assigned role codes. Used by auth middleware. |
| `getAllApiKeys` | `() → ApiKeyListItem[]` | List all keys with role codes; for admin UI. |
| `getApiKeyById` | `(id: number) → ApiKeyWithRoles \| undefined` | Single key with roles; for admin detail/edit. |
| `createApiKey` | `(data: InsertApiKey, roleCodes: string[]) → ApiKey` | Insert key record and assign roles in `api_key_roles`. |
| `updateApiKey` | `(id: number, data: Partial<ApiKey>) → ApiKey` | Update name, description, isActive, expiresAt. |
| `updateApiKeyRoles` | `(id: number, roleCodes: string[]) → void` | Replace all role assignments for a key (delete existing, insert new). |
| `deleteApiKey` | `(id: number) → void` | Hard delete (cascades to `api_key_roles`). |
| `touchApiKeyLastUsed` | `(id: number) → void` | Update `last_used_at` to `now()`. |

### Types

```typescript
type ApiKeyWithRoles = ApiKey & { roles: string[] };
type ApiKeyListItem = ApiKey & { roles: string[], createdByName: string | null };
```

## 3. Auth Middleware — Dual-Path Authentication

Modify `requireAuth` in `server/lib/auth.ts` to accept **either** a session cookie (existing) **or** an API key via Bearer token.

### The `req.user` Contract

Currently, session auth populates `req.user` with a shape like `{ id, email, displayName, roles, isActive }`. API key auth must populate `req.user` with a **compatible shape** so that downstream middleware (`requirePermission`, `hasPermission`) works without branching:

```typescript
// Session-authenticated user (existing)
req.user = {
  id: "azure-oid-123",
  email: "jane@leatherman.com",
  displayName: "Jane Smith",
  roles: ["editor"],
  isActive: true,
  authMethod: "session"       // new field
};

// API key-authenticated identity
req.user = {
  id: "apikey:42",            // synthetic ID — prefixed to avoid collisions with user OIDs
  email: null,
  displayName: "ERP Sync",   // the key's name
  roles: ["editor"],          // from api_key_roles
  isActive: true,
  authMethod: "api_key",      // new field
  apiKeyId: 42                // the api_keys.id — useful for audit
};
```

The `authMethod` field lets audit logging and any future logic distinguish between user and API key requests. The `roles` array is resolved at auth time from `api_key_roles`, so `requirePermission` / `hasPermission` works identically — it just reads `req.user.roles`.

### Authentication Flow

```
Incoming request
  │
  ├─ Has `Authorization: Bearer earl_...` header?
  │   ├─ YES → SHA-256 hash the token
  │   │         Look up via getApiKeyByHash()
  │   │         Verify isActive === true
  │   │         Verify expiresAt is null or in the future
  │   │         Build req.user from key record + api_key_roles
  │   │         Fire-and-forget: touchApiKeyLastUsed()
  │   │         → next()
  │   │
  │   └─ Key not found / inactive / expired → 401
  │
  ├─ Has valid session cookie? (existing Passport flow)
  │   └─ YES → req.user already populated by deserializeUser → next()
  │
  └─ Neither → 401 Unauthorized
```

### `hasPermission` Adaptation

The existing `hasPermission(userId, resourceType, action, scope)` from Phase 19 queries `user_roles` → `role_permissions` → `permissions`. Since API key auth resolves roles at auth time and attaches them to `req.user.roles`, the simplest adaptation is:

- **Option A (preferred):** Refactor `requirePermission` to check permissions using `req.user.roles` directly (already loaded) rather than re-querying `user_roles` by user ID. This is a performance improvement for session auth too and makes API key auth work with zero branching.
- **Option B:** If `hasPermission` must keep its current DB-query approach, add a parallel code path that queries `api_key_roles` when `req.user.authMethod === "api_key"`.

Option A is recommended — roles are already resolved and attached to `req.user` during authentication, so there's no reason to hit the DB again in the permission check.

### Applying to Both REST and GraphQL

The middleware must be applied to:

- **REST:** All `/api/v1/*` routes (already the case).
- **GraphQL:** The `/graphql` endpoint (or whatever path is used). The GraphQL server's context factory should read `req.user` — which will already be populated by the shared middleware — and pass it into the resolver context. This means API key auth works for GraphQL with zero additional resolver changes.

### Error Responses

Follow Phase 17 error format:

```json
{
  "error": {
    "code": "UNAUTHORIZED",
    "message": "Invalid or expired API key."
  }
}
```

For GraphQL, the same 401 status is returned at the HTTP transport level (before the GraphQL layer processes the query), consistent with how session auth failures are handled.

## 4. API Key Management Endpoints

Admin-only REST endpoints for key lifecycle. All require `requireAuth` + admin role check.

### `GET /api/v1/api-keys`

List all API keys. Returns metadata only — never the key itself.

```json
{
  "data": [
    {
      "id": 1,
      "name": "ERP Sync",
      "description": "Nightly product sync from SAP",
      "keyPrefix": "earl_a3f8",
      "roles": ["editor"],
      "isActive": true,
      "expiresAt": "2026-12-31T00:00:00Z",
      "lastUsedAt": "2026-03-02T14:30:00Z",
      "createdBy": "Jane Smith",
      "createdAt": "2026-01-15T10:00:00Z"
    }
  ],
  "meta": { "total": 1 }
}
```

### `POST /api/v1/api-keys`

Create a new API key. The full plaintext key is returned **only in this response** — it cannot be retrieved again.

**Request:**

```json
{
  "name": "ERP Sync",
  "description": "Nightly product sync from SAP",
  "roles": ["editor"],
  "expiresAt": "2026-12-31T00:00:00Z"
}
```

**Validation:**
- `name` is required, non-empty.
- `roles` is required, non-empty array. Each role code must exist in the `roles` table.
- `expiresAt` is optional. If provided, must be a future date.

**Response (201):**

```json
{
  "data": {
    "id": 1,
    "name": "ERP Sync",
    "description": "Nightly product sync from SAP",
    "keyPrefix": "earl_a3f8",
    "key": "earl_a3f8b2c1d4e5f67890abcdef12345678",
    "roles": ["editor"],
    "isActive": true,
    "expiresAt": "2026-12-31T00:00:00Z",
    "createdAt": "2026-01-15T10:00:00Z"
  }
}
```

**Server-side flow:** Generate 32 random hex bytes → prepend `earl_` → SHA-256 hash → store hash and prefix in `api_keys` → insert role assignments in `api_key_roles` → return plaintext key once.

### `PATCH /api/v1/api-keys/:id`

Update name, description, isActive, expiresAt, and/or roles. If `roles` is provided, it replaces the full set of role assignments (via `updateApiKeyRoles`).

**Request (all fields optional):**

```json
{
  "name": "ERP Sync — Production",
  "isActive": false,
  "roles": ["viewer"]
}
```

### `DELETE /api/v1/api-keys/:id`

Hard delete a key. Returns `204 No Content`.

### Swagger

Add all four endpoints to `server/swagger.ts` with request/response schemas, following Phase 17 patterns.

## 6. Audit Trail

Since API keys are not linked to a user, audit history needs a small adaptation:

- **`authMethod` on audit records:** If audit history records currently store only a `userId`, extend the audit record (or its metadata) to also capture `authMethod` (`"session"` or `"api_key"`) and optionally `apiKeyId` / `apiKeyName`. This way the audit trail reads "Changed by **ERP Sync** (API key)" rather than attributing actions to a user who doesn't exist.
- **Display:** In the audit history UI (Phase 20), if `authMethod === "api_key"`, display the key name with a key icon instead of a user avatar/name.
- **Key deletion:** If an API key is deleted, its past audit entries should still display the key name. Store the key name in the audit record at write time (denormalized), not just a foreign key reference.

## 7. GraphQL Considerations

If/when a GraphQL endpoint is added to EARL:

- **Auth is already handled** by the shared `requireAuth` middleware at the HTTP layer (Section 3). The GraphQL context factory reads `req.user` and passes it to resolvers.
- **Resolvers call `requirePermission`** (or the equivalent `hasPermission` check) the same way REST route handlers do — the identity shape is identical regardless of whether auth came from a session or an API key.
- **No separate GraphQL auth scheme** is needed. API keys work for GraphQL out of the box because authentication happens before the request reaches the GraphQL execution layer.
- **Rate limiting and usage tracking** (if added later) should apply uniformly to both REST and GraphQL requests authenticated via API key.

## 9. Security Considerations

- **Key storage:** Only the SHA-256 hash is stored; plaintext is shown once at creation and never persisted.
- **Key prefix:** The `key_prefix` (first 8 chars) is stored for identification in the UI and logs without exposing the full key.
- **Least privilege:** Each key should be assigned only the roles it needs. Avoid granting admin to integration keys — use editor or viewer as appropriate.
- **Rotation:** To rotate, create a new key with the same roles, update the consuming system, then delete the old key. No in-place rotation — this avoids the risk of a window where neither key works.
- **Expiry:** Optional but recommended for external integrations. The admin UI should surface approaching expirations.
- **Revocation:** Setting `isActive = false` immediately blocks authentication; delete for permanent removal.
- **Rate limiting:** Not in scope for this phase, but the `last_used_at` tracking lays groundwork for future rate limiting or anomaly detection.
- **Audit:** Key creation, modification, role changes, and deletion should themselves be logged in audit history.

## 11. Notes

- **Why API keys over OAuth 2.0 Client Credentials?** Phase 12.4 proposed full OAuth 2.0 client credentials via Entra ID. API keys are simpler: no additional Entra app registrations, no token endpoint calls, no token refresh logic for consumers. For internal and trusted integrations (ERP, e-commerce, import scripts), API keys with RBAC are sufficient. If a future integration requires OAuth 2.0 (e.g. a third-party SaaS that only supports OAuth), that can be added as a separate phase.
- **Why not link to users?** API keys are standalone identities so that keys survive user account changes (deactivation, departure) and so that integrations aren't incorrectly attributed to a human. Keys get their own role assignments for independent permission scoping.
- **Key naming convention:** Encourage descriptive names that identify the consuming system and environment (e.g. "ERP Sync — Production", "Shopify Staging").
- **No new dependencies.** Uses Node.js built-in `crypto` for SHA-256 hashing and random key generation. No external auth libraries needed beyond what Phase 01 already installed.
