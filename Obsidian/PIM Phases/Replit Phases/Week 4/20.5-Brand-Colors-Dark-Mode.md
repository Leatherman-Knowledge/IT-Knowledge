## 2. Accessibility Contrast Matrix

From the brand style guide's contrast testing. **Do not use combinations marked FAIL for text.**

| Text Color | Background | Approx. Ratio | WCAG AA | Rule |
|-----------|-----------|---------------|---------|------|
| Slate Gray (#393B42) | White | ~9.5:1 | ✅ Pass | Body text on light backgrounds |
| White | Slate Gray | ~9.5:1 | ✅ Pass | Inverse text on dark/branded surfaces |
| Blue (#82B4BD) | White | ~2.5:1 | ❌ Fail | **Use for accents only; not body text on white** |
| Slate Gray | Yellow | ~4.4:1 | ❌ Fail | **Do not use** |
| Yellow | White | ~1.6:1 | ❌ Fail | **Never use Yellow as a text color** |
| Slate Gray | Blue | ~4:1 | ⚠️ Large text only | Use with caution |
| Blue | Slate Gray | ~1.5:1 | ❌ Fail | **Do not use** |

### Key Implications for Components

1. **Yellow (primary) buttons:** Must use near-black text (`#171612` / `50 16% 8%`), **not** Slate Gray. Slate Gray on Yellow fails WCAG AA at ~4.4:1.
2. **Yellow as text:** Never use Yellow as a text color on any background — it fails against White, Gray 50, and Gray 200.
3. **Blue (#82B4BD):** This is a lighter, muted blue. Use for links and accents on White or light gray; avoid as body text on white (contrast is insufficient). For link text, consider Slate Gray with an underline or the blue only on hover if needed.
4. **Dark sidebar:** Use White/near-white text on Slate Gray (#393B42) background — this passes at ~9.5:1.

## 4. Dark Mode Toggle

### ThemeProvider

**File:** `client/src/components/theme-provider.tsx`

Wrap the entire app in a `ThemeProvider` that manages theme state via React context.

| Behavior | Details |
|----------|---------|
| **Three states** | `light`, `dark`, `system` |
| **Default** | `system` — respects the user's OS `prefers-color-scheme` |
| **Persistence** | `localStorage` key `"pim-theme"` |
| **Mechanism** | Add/remove the `dark` class on `<html>` element (Tailwind `darkMode: "class"` strategy) |
| **System listener** | `matchMedia("(prefers-color-scheme: dark)")` change event updates the resolved theme in real time when set to `system` |
| **Context shape** | `{ theme: "light" \| "dark" \| "system", setTheme: (t) => void, resolvedTheme: "light" \| "dark" }` |

### ThemeToggle

**File:** `client/src/components/theme-toggle.tsx`

| Behavior | Details |
|----------|---------|
| **Location** | App header/topbar, right side (near user menu) |
| **UI** | Dropdown button showing the current state icon. Items: ☀️ Light, 🌙 Dark, 🖥 System |
| **Icons** | `Sun`, `Moon`, `Monitor` from lucide-react |
| **Transition** | Apply `transition-colors duration-200` on `<body>` so the theme switch animates smoothly rather than snapping |
| **Accessibility** | `aria-label="Toggle theme"`, dropdown items are labeled |

### FOUC Prevention

Add an inline `<script>` in `index.html` **before** any CSS or React bundles:

```html
<script>
  (function() {
    var t = localStorage.getItem("pim-theme");
    var dark = t === "dark" || (!t || t === "system") &&
      window.matchMedia("(prefers-color-scheme: dark)").matches;
    if (dark) document.documentElement.classList.add("dark");
  })();
</script>
```

This runs synchronously before first paint, preventing a flash of the wrong theme.

### Tailwind Config

Ensure `tailwind.config.ts` has:

```ts
export default {
  darkMode: "class",
  // ...
}
```

## 5. Component Migration Notes

Most shadcn/ui components inherit from CSS custom properties and will pick up the new tokens automatically. The following require explicit attention:

### Sidebar

The light-mode sidebar uses a branded Slate Gray (#393B42) surface (dark background, white text, Yellow active indicator). Verify the sidebar component reads from `--sidebar-*` tokens and does not use hardcoded colors. The Yellow active item highlight is the key brand touch. **Display the EARL logo** in the sidebar header (see PIM Name & Logo section above).

### Buttons

| Variant | Light Mode | Dark Mode |
|---------|-----------|-----------|
| Primary (default) | Yellow bg, near-black text | Yellow bg, near-black text (unchanged) |
| Secondary | Blue (#82B4BD) bg, white text | Slightly brighter blue bg, white text |
| Destructive | Red bg, white text | Brighter red bg, white text |
| Outline | White bg, Slate Gray (#393B42) text, Gray 200 border | Dark card bg, warm near-white text, dark border |
| Ghost | Transparent, Slate Gray (#393B42) text | Transparent, warm near-white text |

### Audit History Badges

Replace any hardcoded badge colors with token-based Tailwind classes:

| Action | Token mapping |
|--------|--------------|
| Create (green) | `bg-success/10 text-success` → maps to `--success` |
| Update (blue) | `bg-info/10 text-info` → maps to `--info` |
| Delete (red) | `bg-destructive/10 text-destructive` → maps to `--destructive` |
| Reorder (yellow) | `bg-primary/10 text-primary-foreground` or a dedicated muted-yellow |

If the custom utility tokens (`--success`, `--info`, `--success-background`, etc.) are not natively supported by shadcn/ui's Tailwind plugin, extend the Tailwind config:

```ts
theme: {
  extend: {
    colors: {
      success: "hsl(var(--success))",
      info: "hsl(var(--info))",
    }
  }
}
```

### Toast / Alert

Map toast variants to utility feedback tokens: `destructive` uses `--destructive` + `--destructive-background`, and add `success` and `info` variants if they don't exist.

### Forms and Validation

The inline validation error component already uses `text-destructive`. This will automatically adopt the brand error red. No changes needed.

### Tables and Data

Table header backgrounds should use `--muted`, row hover should use `--accent`. Stripe pattern (if used) alternates between `--background` and `--muted`.
