# Phase 25.2: Akeneo Import — Product Attribute Structure

**Execution Plan Item:** #25 (sub-phase 2 of 5)  
**Replit Phase:** 25.2 (Week 6)  
**Dependencies:** Phase 25.1 (connector scaffold, auth, wipe mechanism)  
**Depended on by:** Phase 25.3 (entity types reference PIM attributes), Phase 25.5 (products depend on families)

**Design note:** The Akeneo importer is temporary (migration only). Do not add code columns to PIM tables. Use in-memory maps (Akeneo code → PIM ID) for the duration of each run. See [[25-Akeneo-Import-Consolidated-Summary-For-Replit]].

---

## 1. In-Memory Maps (Run-Scoped)

Build and retain these maps for the duration of the run. They are consumed by later steps in 25.2 and by phases 25.3–25.5.

- **Attribute group code → PIM attribute group ID** — after each attribute group insert.
- **Attribute code → PIM attribute ID** — after each attribute insert.
- **Attribute option code → PIM option ID** (keyed by attribute code) — after each option insert; used in 25.5 for product attribute values.
- **Family code → PIM family ID** — after each family insert; used in 25.5.

---

## Objective

Add the first import step to the Akeneo connector: **product attribute structure**. After the pre-import wipe (from 25.1), the connector now imports Akeneo attribute groups, attributes (with their option sets for select types), and families (with variant definitions). After this phase, a completed run leaves the PIM with a full product schema — every family, attribute, and attribute group from Akeneo — ready for product records to be written in 25.5.

---

## 2. Akeneo API Endpoints Used in This Phase

All endpoints use cursor pagination (`_links.next`). Use `?limit=100`. On HTTP 429, back off exponentially and retry up to 3 times before logging a failure.

| Entity | Endpoint |
|--------|----------|
| Attribute groups | `GET /api/rest/v1/attribute-groups` |
| Attributes | `GET /api/rest/v1/attributes` (filter by `?families=code1,code2` if `scope_families` set) |
| Attribute options | `GET /api/rest/v1/attributes/{code}/options` (per `singleSelect`/`multiSelect` attribute) |
| Families | `GET /api/rest/v1/families` (filter by codes if `scope_families` set) |
| Family variants | `GET /api/rest/v1/families/{code}/variants` (per family) |

---

## 3. Import Step 1 — Attribute Groups

Fetch all attribute groups from Akeneo and create them in the PIM.

- Map **labels only** to PIM attribute group label (first available locale, or Akeneo code as fallback). Do not add or use a PIM attribute group `code` column.
- After each insert, store **Akeneo group code → PIM attribute group ID** in the run’s in-memory map.
- Log each group as `success` or `failure`.

---

## 4. Import Step 2 — Attributes

Fetch all attributes (filtered by `scope_families` if set). For each attribute:

**Attribute type mapping:**

| Akeneo Type | PIM Type | Notes |
|---|---|---|
| `pim_catalog_text` | `text` | |
| `pim_catalog_textarea` | `textarea` | |
| `pim_catalog_boolean` | `boolean` | |
| `pim_catalog_number` | `number` | |
| `pim_catalog_metric` | `measurement` | Import unit and value; preserve unit family if possible. |
| `pim_catalog_price_collection` | `price` | Multi-currency; import all currencies. |
| `pim_catalog_date` | `date` | ISO 8601 format. |
| `pim_catalog_simpleselect` | `singleSelect` | Also import option set (see below). |
| `pim_catalog_multiselect` | `multiSelect` | Also import option set (see below). |
| `pim_catalog_file` | `assetSingle` | Attribute definition created; binary values are not imported (logged as skip in 25.5). |
| `pim_catalog_image` | `assetSingle` | Same as file. |
| `pim_catalog_asset_collection` | `assetList` | Linked to an asset type; asset types are imported in 25.4. |
| `pim_reference_data_simpleselect` | `entitySingle` | Linked to an entity type; entity types are imported in 25.3. |
| `pim_reference_data_multiselect` | `entityList` | Multi-value entity reference; same entity type link as above. |
| `pim_catalog_table` | `table` | Import if table structure maps cleanly; otherwise log skip with reason. |
| Unknown / unmapped | — | Log skip: `"Unsupported Akeneo attribute type: {type}"`. |

**Per attribute, map:**
- **Labels** → PIM attribute label (first available locale, or Akeneo code as fallback). Do not map to a PIM attribute `code` column.
- `type` → PIM attribute `type` (via table above)
- `group` → PIM attribute group **ID** (resolve via in-memory attribute group map; log skip if group not found)
- `localizable` → PIM attribute `localizable` flag
- `scopable` → PIM attribute `scopable` (channelizable) flag
- `unique` → PIM attribute `unique` flag (validation rule)
- After insert, store **Akeneo attribute code → PIM attribute ID** in the run’s in-memory map.

**Attribute options** (for `singleSelect` and `multiSelect` only):
- After creating the attribute, fetch `GET /api/rest/v1/attributes/{code}/options` (paginate all pages).
- Create each option: **labels** → PIM option label (no PIM option code column). After each insert, store **Akeneo option code → PIM option ID** in the run’s option map for this attribute (for 25.5 value resolution).
- Log as part of the attribute's record entry (e.g., include option count in `details` JSON rather than separate log rows).

Log each attribute as `success`, `failure`, or `skip`.

---

## 5. Import Step 3 — Families

Fetch all families (filtered by `scope_families` if set — affects API fetch only; wipe in 25.1 is always full). For each family:

**Family mapping:**
- **Labels** → PIM family label (first available locale, or Akeneo code as fallback). Do not add or use a PIM family `code` column.
- `attributes` array → PIM `family_attributes` rows: resolve each Akeneo attribute code to PIM attribute **ID** via the in-memory attribute map; skip unresolved with a log.
- `attribute_requirements` → PIM `family_attribute_requirements` per channel
- After insert, store **Akeneo family code → PIM family ID** in the run’s in-memory map.

**Family variant definitions:**
- Fetch `GET /api/rest/v1/families/{code}/variants` for each family.
- Each Akeneo family variant becomes a PIM `FamilyVariantDefinition` with axis attributes and level assignments.
- Akeneo supports up to 2 variant levels; map level 1 axes to level 1, level 2 axes to level 2.
- If a family has no variants, create a zero-axis variant definition (top model only).
- Log any family where an axis attribute code cannot be resolved (via in-memory attribute map) as a skip with reason.

Log each family as `success`, `failure`, or `skip`.

---

## 6. Run Execution — This Phase

Extend the Phase 25.1 connector. After the wipe and before completing the run, execute these three steps in order:

1. Import attribute groups (§3).
2. Import attributes with options (§4).
3. Import families with variant definitions (§5).

Per-record logging for every attribute group, attribute, and family via `appendIntegrationRunLog`.

**Error handling:**
- Single-record failure: log as `failure`, continue to next record.
- Transient network error: retry up to 2 times (exponential backoff); log `failure` if retries exhausted.
- Unrecoverable step error (e.g., Akeneo returns 500 on all retries for a full page): abort remaining steps, set run `status: 'failed'` with error message indicating which step failed.
- Validation errors writing to PIM: log as `failure`, do not abort the run.

Because the wipe (25.1) ran first, **insert** all records. Do not implement code-based upsert; the PIM has no code columns for these entities. If `wipe_before_import` is ever disabled, log a warning that incremental import is not supported. See [[25-Akeneo-Import-Consolidated-Summary-For-Replit]].

---

## 7. Exit Criteria

- [ ] After a successful run, all Akeneo attribute groups exist in the PIM with correct labels (no code column).
- [ ] All Akeneo attributes exist in the PIM with the correct type, group assignment (by ID), localizable/scopable flags, and labels.
- [ ] `singleSelect` and `multiSelect` attributes have their full option sets imported (labels only; option map populated for 25.5).
- [ ] `assetSingle`, `assetList`, and `entitySingle`/`entityList` attribute definitions are created with the correct type; no values are populated yet (that comes in 25.3–25.5).
- [ ] All Akeneo families exist in the PIM with correct labels, attribute assignments (by attribute ID), required-attribute rules, and family variant definitions (including zero-axis for families with no variants). Family map populated for 25.5.
- [ ] Attributes that could not be resolved to a PIM type are logged as `skip` with reason; the run continues.
- [ ] `scope_families` correctly limits which attributes and families are **fetched from the Akeneo API** (wipe in 25.1 is always full).
- [ ] Run history in Phase 24 UI shows per-record log entries for every imported attribute group, attribute, and family.
