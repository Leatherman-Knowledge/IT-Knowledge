# Phase 25.4: Akeneo Import — Asset Types & Assets

**Execution Plan Item:** #25 (sub-phase 4 of 5)  
**Replit Phase:** 25.4 (Week 6)  
**Dependencies:** Phase 25.3 (entity types and entities imported — PIM now has full entity schema)  
**Depended on by:** Phase 25.5 (products with `assetList`/`assetSingle` values depend on assets existing)

**Design note:** No PIM code columns. Use in-memory maps: Akeneo asset family (type) code → PIM asset type ID; Akeneo asset code → PIM asset ID (per type). See [[25-Akeneo-Import-Consolidated-Summary-For-Replit]].

---

## Objective

Add asset type and asset import to the Akeneo connector. Akeneo **Asset Manager** (Enterprise Edition only, new-style — not the legacy PAM) maps to PIM asset types and asset records. After this phase, asset types with their full attribute schemas and all asset records exist in the PIM. Product `assetList` and `assetSingle` attribute values written in Phase 25.5 can then resolve to real asset IDs.

This step is **gracefully skipped** if the Akeneo instance does not support the Asset Manager API — the connector logs one info entry and continues without error.

---

## 1. EE Availability Check

At the start of the asset type import step, attempt `GET /api/rest/v1/asset-families`.

- **404 or 403 response:** Log a single `skip` entry:
  ```
  recordIdentifier: "asset-types-import"
  status: "skip"
  message: "Asset Manager not available on this Akeneo instance (CE, legacy PAM, or unsupported version). Skipping asset type and asset import."
  ```
  Continue to Phase 25.5 (categories and products). Do not abort the run.
- **200 response:** Proceed with the import below.

---

## 2. Akeneo API Endpoints Used in This Phase

| Entity | Endpoint |
|--------|----------|
| Asset families (types) | `GET /api/rest/v1/asset-families` |
| Asset family attributes | `GET /api/rest/v1/asset-families/{code}/attributes` |
| Asset family attribute options | `GET /api/rest/v1/asset-families/{code}/attributes/{attributeCode}/options` |
| Asset records | `GET /api/rest/v1/asset-families/{code}/assets` |

All endpoints paginate via `_links.next`. Use `?limit=100`. Retry on HTTP 429 with exponential backoff (up to 3 retries).

---

## 3. Import Step 5a — Asset Types (Asset Family Definitions)

For each asset family returned by `GET /api/rest/v1/asset-families`:

- Map **labels** → PIM asset type label (first available locale, or Akeneo code as fallback). Do not add or use a PIM asset type `code` column.
- After each insert, store **Akeneo asset family code → PIM asset type ID** in the run’s in-memory map (for 25.5 product `assetList`/`assetSingle` resolution).
- Fetch attributes for this asset family (`GET /api/rest/v1/asset-families/{code}/attributes`).
- Create the asset type in PIM, then create each attribute definition on it.

**Asset family attribute type mapping:**

| Akeneo Asset Attribute Type | PIM Type | Notes |
|---|---|---|
| `text` | `text` | |
| `textarea` | `textarea` | |
| `number` | `number` | |
| `boolean` | `boolean` | |
| `date` | `date` | |
| `media_file` | `assetSingle` | The primary asset file. Attribute definition is created; binary values are not migrated in MVP (logged as skip in record import). |
| `single_option` / `option` | `singleSelect` | Fetch and import option list. |
| `multiple_options` | `multiSelect` | Fetch and import option list. |
| `url` | `text` | |
| Unknown | — | Log skip: `"Unsupported asset attribute type: {type}"`. |

For `single_option` / `multiple_options`, fetch options from `GET /api/rest/v1/asset-families/{code}/attributes/{attributeCode}/options` and create the PIM option set.

Log each asset type as `success`, `failure`, or `skip`.

---

## 4. Import Step 5b — Asset Records

After all asset types are created, import assets for each asset family:

Fetch `GET /api/rest/v1/asset-families/{code}/assets` (paginated). For each asset:

- Map **identifier**: use Akeneo `code` as the PIM asset identifier (or the field the PIM uses for external reference). Do not add a PIM asset `code` column.
- After each insert, store **Akeneo asset code → PIM asset ID** in the run’s asset map for this asset type (for 25.5 product `assetList`/`assetSingle` value resolution).
- Set asset type → the PIM asset type created in step 5a (via in-memory map).
- Map `values` → PIM asset attribute values using locale/channel resolution:
  - Akeneo locale → PIM locale. If not found, log skip for that value with reason `"Locale {locale} not found in PIM"`.
  - `media_file` attribute values (binary files) — log skip: `"Binary asset file not migrated in Phase 25."` The PIM asset record is still created; only the binary value is skipped.
- Log each asset record as `success`, `failure`, or `skip`.

---

## 5. Run Execution — This Phase

Extend the Phase 25.3 connector. After importing entity records, add these two steps before completing the run:

**Step 5a:** Import asset types (asset family definitions + attribute schemas).  
**Step 5b:** Import asset records (with attribute values, binary files skipped).

These steps run after entity types/entities (25.3) and before categories/products (25.5).

Same error handling as previous sub-phases: per-record failures continue the run; unrecoverable step failures abort with `status: 'failed'` naming the failed step.

---

## 6. Exit Criteria

- [ ] If Asset Manager is unavailable: a single `skip` log entry appears, the run continues, and no asset types or assets are created. No error.
- [ ] If Asset Manager is available: all Akeneo asset families exist in the PIM with correct labels and attribute definitions (no code column). Asset type and asset maps populated for 25.5.
- [ ] `singleSelect`/`multiSelect` asset type attributes have their option sets imported.
- [ ] All asset records exist in the PIM as assets of the correct asset type, with correct non-binary attribute values.
- [ ] `media_file` attribute values are skipped with a logged reason (`"Binary asset file not migrated in Phase 25."`); the asset record itself is still created.
- [ ] Per-record log entries are present for every asset type and asset record in the Phase 24 run logs UI.
- [ ] Attribute resolution failures (unknown type) are logged as `skip`; the run does not abort.

---

## 7. Notes

- **Binary files are not migrated in this phase.** Asset records will exist in the PIM with all non-binary attribute values. Media migration is a follow-on task outside the MVP scope.
- **Asset records before products.** Asset records must be fully imported here so that Phase 25.5 can resolve `assetList`/`assetSingle` values on products to PIM asset IDs via the run’s asset maps.
- **Legacy Akeneo PAM.** Older Akeneo instances use a different asset system (PAM) with different endpoints. If the Asset Manager endpoint returns 404 and there is evidence of a PAM setup, log the skip message as specified — PAM migration is out of scope for this phase.
