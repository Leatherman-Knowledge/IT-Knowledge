# Phase 25.5: Akeneo Import — Categories, Products & Variants

**Execution Plan Item:** #25 (sub-phase 5 of 5)  
**Replit Phase:** 25.5 (Week 6)  
**Dependencies:** Phase 25.4 (asset types and assets imported — all referenced data exists in PIM)  
**Depended on by:** Phase 26 (import monitoring & verification), Phase 27 (GraphQL — real data), Phase 28 (Search — real data)

**Design note:** The only codes stored in the PIM are **product identifier** and **SKU**. All other resolution (family, variant definition, parent, categories, attributes, options, entities, assets) uses the run’s in-memory maps from 25.2–25.4. No PIM code columns for categories. See [[25-Akeneo-Import-Consolidated-Summary-For-Replit]].

---

## Objective

Complete the Akeneo Import connector by adding the final two import steps: **categories** and **products with full variant hierarchies and attribute values**. After this phase, the PIM is a complete mirror of Akeneo — every category tree, product model, variant, and leaf product exists with all their attribute values, status, and category assignments. The full 9-step connector is now operational.

This is the largest single step in the import pipeline. All prerequisite data (attribute structure from 25.2, entities from 25.3, assets from 25.4) is already in the PIM, so `entityReference` and `assetList`/`assetSingle` values on products can resolve to real IDs.

## 1. Akeneo API Endpoints Used in This Phase

| Entity | Endpoint |
|--------|----------|
| Categories | `GET /api/rest/v1/categories` |
| Product models (parents) | `GET /api/rest/v1/product-models` |
| Products (leaves + simples) | `GET /api/rest/v1/products` |

All endpoints paginate via `_links.next`. Use `?limit=100`. Respect rate limits (HTTP 429) with exponential backoff and retry up to 3 times.

---

## 2. Import Step 6 — Categories

Akeneo returns all categories as a flat list with a `parent` field (null for root-level categories).

**Import process:**
1. Fetch all categories in one pass, building an in-memory map of `Akeneo category code → { label, parent }`.
2. Sort topologically by depth (root categories first, then children) to ensure parents exist before children are written.
3. For each category, map: **labels** → PIM category label (first available locale, or Akeneo code as fallback). Do not add or use a PIM category `code` column. Resolve `parent` → PIM parent category **ID** via the category map (after parent is inserted). Store **Akeneo category code → PIM category ID** in the run’s category map for product–category assignment in step 7.
4. Root-level categories (null parent): if no root category tree exists yet, create one named "Akeneo" and assign all root-level Akeneo categories under it. If a tree/root already exists with a matching structure, use it.

Log each category as `success`, `failure`, or `skip`.

## 3. Import Step 7 — Products and Variants

Akeneo products come in two forms:

- **Product models** (`/api/rest/v1/product-models`): parent records that define a variant hierarchy. Have a `parent` field (null for root models) and a `family_variant` field.
- **Products** (`/api/rest/v1/products`): leaf products with a `parent` pointing to a product model, or null for simple (no-variant) products.

### 3.1 Import Order Within Products

Fetch and import in this sequence to ensure parents exist before children:

1. **Root product models** — `parent == null`. These become PIM parent products.
2. **Child product models** — `parent != null`, ordered by depth (shallowest first). These become PIM intermediate variant products.
3. **Leaf products** — products with a `parent` pointing to a product model. These become PIM variant leaf products.
4. **Simple products** — products with `parent == null` (not product models). These become PIM top-model products with a zero-axis variant definition.

Apply filters: if `scope_channel` is set, pass `?scope={scope_channel}` on product endpoints. If `scope_families` is set, pass `?families={scope_families}` on products.

### 3.2 Product Mapping

| Akeneo Field | PIM Mapping |
|---|---|
| `identifier` / `code` | PIM product **identifier** (and **SKU** where applicable). These are the only codes persisted in the PIM. |
| `family` | Resolve to PIM family **ID** via the run’s family map (from 25.2). Log skip if not found. |
| `family_variant` | Resolve to PIM `FamilyVariantDefinition` ID (imported in 25.2; resolve via family + variant info from 25.2). |
| `parent` | Resolve to PIM parent product ID from the run’s product map (parents inserted earlier in this step). Log failure if parent not yet created. |
| `enabled: true` | PIM `status: 'active'` |
| `enabled: false` | PIM `status: 'draft'` |
| `categories` array | PIM product–category assignments. Resolve each Akeneo category code to PIM category **ID** via the run’s category map. Skip any code not in map with a log entry. |

### 3.3 Attribute Values

For each product or product model, iterate its `values` map from the Akeneo response. For each attribute code + locale + scope combination:

- **Attribute lookup:** Resolve the Akeneo attribute code to a PIM attribute **ID** using the run’s attribute map (from 25.2). If Phase 22 field mappings are configured, map source (Akeneo code) to target PIM attribute **ID** (from map). Log skip with reason `"No PIM attribute found for Akeneo code: {code}"` if unresolvable.
- **Locale resolution:** Map Akeneo locale to PIM locale. Log skip for the value (not the whole product) if not found: `"Locale {locale} not found in PIM"`.
- **Channel resolution:** Map Akeneo scope to PIM channel. Log skip for the value if not found: `"Channel {scope} not found in PIM"`.
- **Non-localizable, non-scopable values:** Store with `locale = null`, `channel = null`.
- **Scope filter:** If `scope_locales` is set, skip values for locales not in the scope list.

**Special value handling by type:**

| PIM Type | Handling |
|---|---|
| `text`, `textarea`, `number`, `boolean`, `date`, `measurement`, `price` | Write directly. |
| `singleSelect`, `multiSelect` | Resolve Akeneo option code to PIM option **ID** via the run’s option map (from 25.2). Log skip if option not found. |
| `entityReference` | Resolve Akeneo reference entity record code to PIM entity **ID** via the run’s entity map (from 25.3). Log skip if entity not found. |
| `assetList`, `assetSingle` | Resolve Akeneo asset code to PIM asset **ID** via the run’s asset map (from 25.4). Log skip if asset not found. |
| `assetSingle` (from `pim_catalog_file` / `pim_catalog_image`) | Binary files are not migrated. Log skip: `"Binary asset file not migrated in Phase 25."` |
| `table` | Import if structure maps cleanly; otherwise log skip. |

Log each product/product model as a single `success`, `failure`, or `skip` entry. Include counts of skipped values in `details` JSON rather than separate log rows (to keep the log manageable for large catalogs). After each product insert, store **Akeneo product identifier/code → PIM product ID** in the run’s product map for parent resolution of subsequent records.

## 4. Field Mapping Support

The connector uses field mappings configured via the Phase 22 UI and storage layer:

- **Default (no mappings configured):** Akeneo attribute code → PIM attribute **ID** via the run’s attribute map (from 25.2).
- **Explicit mappings configured:** `source` = Akeneo attribute code, `target` = PIM attribute **ID** (resolve from the run’s attribute map; mappings may alias one Akeneo code to a different PIM attribute).
- Load all field mappings for the instance at the start of the run (before the wipe). Cache them in memory for the duration of the run. Resolution is always to PIM attribute ID, not code.

## 5. Full Run Execution — Complete Connector

With 25.5 complete, the full Akeneo Import connector run executes all 9 steps:

1. Authenticate with Akeneo.
2. Pre-import wipe (if `wipe_before_import` is true).
3. Import attribute groups.
4. Import attributes (with option sets).
5. Import families (with variant definitions).
6. Import entity types + entity records (EE; graceful skip if unavailable).
7. Import asset types + asset records (EE; graceful skip if unavailable).
8. Import categories.
9. Import products (root models → child models → leaf products → simple products) with all attribute values.

All steps log per-record entries via the Phase 23 run engine contract. The run completes with `status: 'completed'` or `status: 'failed'` depending on whether any step aborted.

**Error handling:**
- Per-record failure: log and continue.
- Transient network error per record: retry up to 2 times; log `failure` if all retries exhausted.
- Unrecoverable step failure: abort remaining steps, set run `status: 'failed'` with `error_message` indicating the step.
- Validation error writing to PIM: log `failure`, continue.

## 6. Exit Criteria

- [ ] All Akeneo categories exist in the PIM in the correct tree hierarchy (parents before children), with labels only (no PIM category code column). Category map populated for product assignments.
- [ ] All Akeneo product models (root and child) exist in the PIM as parent/intermediate products with correct family ID, variant definition, and parent–child relationships. Product identifier and SKU are the only codes stored.
- [ ] All Akeneo leaf products exist in the PIM under the correct parent with correct axis attribute values.
- [ ] Simple products (no Akeneo parent) exist as standalone top-model products.
- [ ] Product status is `active` for enabled products and `draft` for disabled products.
- [ ] Product category assignments use PIM category IDs from the category map; Akeneo `categories` array resolved correctly.
- [ ] Attribute values are correctly written: attribute and option resolution use run maps (attribute ID, option ID); `entityReference` and `assetList`/`assetSingle` use entity and asset maps.
- [ ] Binary `assetSingle` values (from `pim_catalog_file`/`pim_catalog_image`) are logged as skip; the product itself is still created.
- [ ] Locale/channel values outside `scope_locales`/`scope_channel` are filtered; in-scope values are written correctly.
- [ ] Field mappings from Phase 22 (if configured) override the default Akeneo-code → PIM-attribute-ID resolution; target is always PIM attribute ID from the run’s map.
- [ ] Per-record log entries exist for every category, product model, and leaf product in the Phase 24 run logs UI.
- [ ] Running the full import a second time produces a clean re-import: the wipe clears all prior data and the fresh import repopulates it with no duplicates or stale records.
- [ ] Real data is confirmed queryable via PIM REST API (`GET /api/v1/products`, `GET /api/v1/families`, etc.) and ready for Phase 27 (GraphQL) and Phase 28 (Search) testing.

---

## 7. Notes

- **Pagination for large catalogs.** Products are the largest dataset. Do not buffer all pages in memory — process and log page-by-page. A full catalog import may return thousands of pages.
- **Parent resolution within a run.** Root product models must be inserted before their children. Maintain a product map (Akeneo identifier/code → PIM product ID) and resolve parents from it. If a leaf product references a parent that hasn’t been created yet (ordering issue), log as `failure` with reason `"Parent product not found: {code}"` and continue.
- **Run duration.** A large catalog (10,000+ products) may take minutes. If the synchronous run model from Phase 23 proves too slow (timeout, UI hang), upgrade to async execution (202 Accepted + polling). Document if this occurs.
- **Phase 26** is the verification pass that confirms end-to-end correctness and the wipe-and-replace cycle.
