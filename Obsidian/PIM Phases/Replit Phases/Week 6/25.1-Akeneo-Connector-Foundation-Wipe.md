**Execution Plan Item:** #25 (sub-phase 1 of 5)

**Dependencies:** Phase 21 (Integration type registry & instances), Phase 22 (Field mappings), Phase 23 (Run engine & logging), Phase 24 (Integration Management UI), Phase 20.6 (API Key Auth)

**Design note:** This connector is temporary — it exists only to support migration from Akeneo. Once migration is complete it will be removed. Do not add permanent schema changes for it.

---

## Objective

Establish the complete **Akeneo Import connector scaffold**: register the integration type, define the connection schema, implement Akeneo OAuth2 authentication, and build the **pre-import data wipe** mechanism. After this phase, triggering a full run on an Akeneo Import instance will authenticate against Akeneo and execute the wipe — but import no data yet (import steps are added in 25.2–25.5). This makes auth and wipe independently verifiable before any import logic is layered on top.

## 3. Akeneo API Authentication

Akeneo uses OAuth2 Password Grant.

**Token request:**

```
POST {base_url}/api/oauth/v1/token
Authorization: Basic base64(client_id:client_secret)
Content-Type: application/json

{ "grant_type": "password", "username": "...", "password": "..." }
```

Response: `{ "access_token": "...", "expires_in": 3600, "token_type": "bearer" }`

**Implementation rules:**
- Acquire the token at the very start of every run, before the wipe.
- Store the token in memory only for the duration of the run; never persist it.
- Use `Authorization: Bearer {access_token}` on all subsequent Akeneo API calls.
- On auth failure (401, network error, timeout): set run `status: 'failed'`, `error_message: "Akeneo authentication failed: {reason}"`. Log no per-record entries — the run-level error is sufficient. **Do not execute the wipe if auth fails.**

**Optional connection test endpoint:**

```
POST /api/v1/integration-instances/:id/test-connection
```

Attempts auth and returns `{ success: true }` or `{ success: false, message: "..." }`. Stub with a successful response if time-constrained; the Phase 24 UI may show a "Test Connection" button that calls it.

## 4. Pre-Import Data Wipe

**This behavior is unique to the Akeneo Import connector. Do not add this logic to the Phase 23 run engine or any other connector.**

When `wipe_before_import` is `true` (the default), the connector deletes all PIM data it manages in a **single database transaction** before any import begins. If the transaction fails, it is rolled back and the run is aborted — no data is deleted or imported.

**Deletion order** (reverse dependency to satisfy FK constraints):

1. Product attribute values
2. Product–category assignments
3. Product associations / relationships
4. Products (leaf-to-root by depth, or via ON DELETE CASCADE if FK cascade is configured)
5. Entity attribute values
6. Entities
7. Asset attribute values
8. Assets
9. Categories (children before parents, or via ON DELETE CASCADE)
10. Family attribute requirements
11. Family attributes
12. Family variant definitions
13. Families
14. Attribute options (product attribute option sets)
15. Product attributes — delete before entity/asset type tables (product attributes may FK to `entity_types` / `asset_types` for `entityReference`/`assetList` types)
16. Attribute groups
17. Entity type attribute options
18. Entity type attributes
19. Entity types
20. Asset type attribute options
21. Asset type attributes
22. Asset types

**What is NOT wiped** — do not touch:
- Channels and locales
- Users, roles, RBAC permissions
- Integration types, integration instances, run history, run logs
- Audit history

**Wipe scope:** The wipe is **always full**. When `wipe_before_import` is true, delete all rows from the above tables in one transaction. Do not implement scope-aware or partial wipe (e.g. do not filter by `scope_families`). The `scope_families`, `scope_locales`, and `scope_channel` config options apply only to **what is fetched from the Akeneo API** in later import steps — they do not affect which data is deleted. See [[25-Akeneo-Import-Consolidated-Summary-For-Replit]].

**Run log entry:** After the wipe transaction commits, log a single entry:

```
recordIdentifier: "pre-import-wipe"
status: "success"
message: "Pre-import wipe complete: deleted {N} products, {P} entities, {Q} assets, {M} families, {K} attributes, {J} categories, {R} entity types, {S} asset types."
```

On transaction failure:

```
recordIdentifier: "pre-import-wipe"
status: "failure"
message: "Pre-import wipe failed: {error}. Run aborted. No data was deleted."
```

Then set run `status: 'failed'` and return.

## 5. Run Execution — This Phase

The connector stub for 25.1 follows the Phase 23 connector contract:

1. Run engine calls connector with `(runId, instanceId)`.
2. `updateIntegrationRun(runId, { status: 'running', started_at: now() })`.
3. Load instance config and scope settings.
4. Authenticate with Akeneo (§3). Abort with `status: 'failed'` on failure.
5. If `wipe_before_import` is true: execute the wipe transaction (§4). Log `pre-import-wipe` entry. Abort with `status: 'failed'` on wipe failure.
6. **No import steps yet** — call `updateIntegrationRun(runId, { status: 'completed', finished_at, record_count_success: 1, record_count_failure: 0, record_count_skip: 0 })` (the wipe entry counts as the one success). Import steps are added in phases 25.2–25.5.

## 7. Exit Criteria

- [ ] `akeneo_import` integration type is seeded and appears in the Integration Types list in the Phase 24 UI with direction "Inbound".
- [ ] Creating an Akeneo Import instance renders the full connection config form (base URL, client_id, client_secret, username, password, optional scope filters, wipe toggle). Secrets are masked after save.
- [ ] Triggering a full run on an instance with **valid** Akeneo credentials: authenticates, executes the wipe transaction, logs the `pre-import-wipe` success entry, and completes the run with `status: 'completed'`.
- [ ] Wipe transaction deletes data from all 22 table categories in the correct order. After the wipe, products, families, attributes, attribute groups, categories, entities, entity types, assets, and asset types tables are empty.
- [ ] Channels, locales, users, RBAC, integration config, audit history are untouched after the wipe.
- [ ] Triggering a run with **invalid** Akeneo credentials: run fails with `status: 'failed'` and a clear auth error message. No data is wiped.
- [ ] Wipe transaction failure (e.g., simulated DB error): run fails, no data is deleted (transaction rolled back), `pre-import-wipe` failure log entry is present.
- [ ] Wipe is always full (all 22 steps, all rows); no scope-aware or partial wipe.
- [ ] No other integration connector type has a wipe mechanism.

## 8. Notes

- **Import steps come in 25.2–25.5.** This phase is deliberately scoped to the scaffold and wipe only. A completed 25.1 run with valid credentials produces exactly one log entry (`pre-import-wipe: success`) and an empty PIM.
- **Auth before wipe, always.** If auth fails, the wipe never runs. The PIM is left untouched.
- **Wipe atomicity.** All 22 deletion steps happen in one transaction. A partial wipe is not possible.
- **Consolidated design.** See [[25-Akeneo-Import-Consolidated-Summary-For-Replit]] for no-code-columns policy, in-memory maps, and temporary-importer context used across 25.1–25.5.
