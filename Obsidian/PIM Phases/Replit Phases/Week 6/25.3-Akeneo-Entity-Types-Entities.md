# Phase 25.3: Akeneo Import — Entity Types & Entities

**Execution Plan Item:** #25 (sub-phase 3 of 5)  
**Replit Phase:** 25.3 (Week 6)  
**Dependencies:** Phase 25.2 (product attribute structure imported — PIM now has attributes and families)  
**Depended on by:** Phase 25.5 (products with `entityReference` attribute values depend on entities existing)

**Design note:** No PIM code columns. Use in-memory maps: Akeneo entity type code → PIM entity type ID; Akeneo entity record code → PIM entity ID (per type). See [[25-Akeneo-Import-Consolidated-Summary-For-Replit]].

---

## Objective

Add entity type and entity import to the Akeneo connector. Akeneo **Reference Entities** (Enterprise Edition only) map to PIM entity types and entity records. After this phase, entity types with their full attribute schemas and all entity records exist in the PIM. Product `entityReference` values written in Phase 25.5 can then resolve to real entity IDs.

This step is **gracefully skipped** if the Akeneo instance is Community Edition or does not support the Reference Entities API — the connector logs one info entry and continues without error.

---

## 1. EE Availability Check

At the start of the entity type import step, attempt `GET /api/rest/v1/reference-entities` (e.g. `?limit=1` to minimize payload).

- **404 or 403 response:** Log a single `skip` entry:
  ```
  recordIdentifier: "entity-types-import"
  status: "skip"
  message: "Reference Entities not available on this Akeneo instance (CE or unsupported version). Skipping entity type and entity import."
  ```
  Continue the run (next steps include 25.4 asset types). Do not abort the run.
- **200 response:** Proceed with the import below.

## 2. Akeneo API Endpoints Used in This Phase

| Entity | Endpoint |
|--------|----------|
| Reference entity definitions | `GET /api/rest/v1/reference-entities` |
| Reference entity attributes | `GET /api/rest/v1/reference-entities/{code}/attributes` |
| Reference entity attribute options | `GET /api/rest/v1/reference-entities/{code}/attributes/{attributeCode}/options` |
| Reference entity records | `GET /api/rest/v1/reference-entities/{code}/records` |

All endpoints paginate via `_links.next`. Use `?limit=100`. Retry on HTTP 429 with exponential backoff (up to 3 retries).

## 3. Import Step 4a — Entity Types (Reference Entity Definitions)

For each reference entity returned by `GET /api/rest/v1/reference-entities`:

- Map **labels** → PIM entity type label (all Akeneo locale keys preserved; if no labels exist, use Akeneo code as fallback under a default key e.g. `en`). Do not add or use a PIM entity type `code` column.
- After each insert, store **Akeneo reference entity code → PIM entity type ID** in the run’s in-memory map (for 25.5 product `entityReference` resolution).
- Fetch attributes for this entity type (`GET /api/rest/v1/reference-entities/{code}/attributes`).
- Create the entity type in PIM, then create each attribute definition on it. Set entity type attributes to `applicableTo: ["entities"]` (distinguishes from product attributes). Entity type attributes are not assigned to attribute groups (`groupId: null`).

**Reference entity attribute type mapping:**

Use namespaced keys in the run’s attribute map to avoid collisions with product attributes: `entity:{refEntityCode}:{attrCode}` for attribute code → PIM attribute ID, and the same namespace for option code → PIM option ID on select attributes.

| Akeneo Reference Entity Attribute Type | PIM Type | Notes |
|---|---|---|
| `text` | `text` | |
| `textarea` | `textarea` | |
| `number` | `number` | |
| `boolean` | `boolean` | |
| `date` | `date` | |
| `image` | `assetSingle` | Attribute definition created; binary values skipped in record import. |
| `file` | `assetSingle` | Same as image. |
| `url` | `text` | |
| `single_option` / `option` | `singleSelect` | Fetch and import option list. |
| `multiple_options` | `multiSelect` | Fetch and import option list. |
| `akeneo_reference_entity` | `entityReference` | Points to another entity type — resolve target code to PIM entity type ID after all entity types are created. |
| `akeneo_reference_entity_collection` | `entityReference` | Multi-value entity reference. |
| Unknown | — | Log skip: `"Unsupported reference entity attribute type: {type}"`. |

For `single_option` / `multiple_options`, fetch options from `GET /api/rest/v1/reference-entities/{code}/attributes/{attributeCode}/options` and create the PIM option set.

For `akeneo_reference_entity` attributes that point to other reference entity types: resolve the target entity type **ID** via the in-memory entity type map (after all entity types have been created — two-pass or deferred resolution). If the referenced entity type is not in the map, log skip for that attribute with reason `"Referenced entity type not found: {code}"`.

Log each entity type as `success`, `failure`, or `skip`.

## 4. Import Step 4b — Entity Records

After all entity types are created, import records for each reference entity type:

Fetch `GET /api/rest/v1/reference-entities/{code}/records` (paginated). For each record:

- Create the PIM entity with `entityTypeId` only (no PIM entity `code` column). Store **Akeneo record code → PIM entity ID** in the run’s entity map for this entity type (for 25.5 product `entityReference` value resolution).
- Map `values` → PIM entity attribute values using locale/channel resolution:
  - Resolve attribute by namespaced key `entity:{refEntityCode}:{attrCode}`. If locale is specified and not found in PIM, log skip for that value with reason `"Locale {locale} not found in PIM for attr \"{attrCode}\""`. If channel is specified, resolve to PIM channel (e.g. by channel label, case-insensitive); if not found, log skip with reason including channel and attr.
  - `image` / `file` attribute values — log skip: `"Binary asset file not migrated in Phase 25."` PIM entity is still created.
  - Use type-aware value mapping: date → `valueDate`, number → `valueNumber`, boolean → `valueBoolean`, select options → resolve option codes to PIM option IDs via run map, text/textarea/url → `valueText`, other complex data → `valueJson`.
- Log each entity record as `success`, `failure`, or `skip`.

## 5. Run Execution — This Phase

Extend the Phase 25.2 connector. After importing families, add these two steps before completing the run:

**Step 4a:** Import entity types (reference entity definitions + attribute schemas).  
**Step 4b:** Import entity records (with attribute values).

These steps run after the wipe and after attribute structure (25.2) so that entity type attributes can reference PIM attribute infrastructure if needed.

Same error handling as 25.2: per-record failures continue the run; unrecoverable step failures abort with `status: 'failed'` naming the failed step.

---

## 6. Exit Criteria

- [ ] If the Akeneo instance is CE (Reference Entities unavailable): a single `skip` log entry appears, the run continues, and no entity types or entities are created. No error.
- [ ] If the Akeneo instance is EE: all reference entity types exist in the PIM with correct labels and attribute definitions (no code column). Entity type map and entity maps populated for 25.5.
- [ ] `singleSelect`/`multiSelect` entity type attributes have their option sets imported.
- [ ] `akeneo_reference_entity` attributes that point to other entity types are resolved to PIM entity type IDs via the in-memory map.
- [ ] All reference entity records exist in the PIM as entities of the correct entity type, with correct attribute values and locale assignments.
- [ ] Binary `image`/`file` values are skipped with a logged reason; the entity record itself is still created.
- [ ] Per-record log entries are present for every entity type and entity record in the Phase 24 run logs UI.
- [ ] Attribute resolution failures (unknown attribute type, missing referenced entity type) are logged as `skip` entries; the run does not abort.

---

## 7. Notes

- **Two-pass entity type attributes:** `akeneo_reference_entity` attributes reference other entity types by code. Resolve to PIM entity type IDs via the in-memory map after all entity types are created (second pass or deferred FK assignment).
- **Entity records before products.** Entity records must be fully imported here so that Phase 25.5 product import can resolve `entityReference` attribute values to PIM entity IDs via the run’s entity maps.
